# Saymore 本地优先语音识别与输入决策地图

状态：当前有效的产品主路线，持续更新；桌面技术栈已决策 日期：2026-07-24

## 1. 终点

产出一套足以进入 PRD 和技术拆票阶段的方向：明确 Saymore
的目标用户、核心价值、首版边界、语音识别架构、风险验证顺序和后续决策。

当前产品只做语音识别、LLM
文本精炼和跨应用文本输入，不做语音助手、问答、回复生成或任务执行。

## 2. 一句话定位

**Saymore
是一个本地优先、模型可替换、能在任何应用中工作的源码可用语音输入工具。**

用户按下快捷键自然说话，Saymore
实时识别语音，并将准确、格式清晰的文字写入当前输入框。

产品链路严格限定为：

```text
说话 -> 语音识别 -> 安全本地清理 -> [LLM 文本精炼] -> 标准写法规范化 -> 文本投递
```

安全本地清理始终执行，只处理首尾空白、异常空格和能够确定由 ASR
技术故障造成的重复片段。用户只控制是否启用 LLM
文本精炼；关闭、未配置或暂时不可用时，系统跳过该阶段。LLM
负责把口语转录保守地整理成忠于原意的书面文本，例如去除无意义填充词和口误重复、处理自我修正、补充标点和整理结构。随后，本地标准写法规范化只把已确认词典中同一
ASCII 字母数字词的大小写和全角/半角形式统一，不推断语义别名。LLM
不回答问题、不扩写用户意图，也不读取当前应用的屏幕内容。

Saymore
的本地模型策略保持克制：团队主动发现适合目标用户的现成模型，完成许可证审查、适配和最小发布验收，然后引导用户从上游官方固定版本下载。Saymore
首版不自建模型 OSS 或镜像，不重新打包上游权重。首个本地 ASR
版本不安排训练、微调或自行量化。模型页的“语音模型”分类固定提供三个本地 ASR
模型：Paraformer、Whisper large-v3-turbo 和 Qwen3-ASR
1.7B；它们不是三个产品方向，也不与文本润色模型混排。产品不建设模型市场、在线目录、社区提交或排行榜。

## 3. 外部产品启发

### 3.1 Typeless 值得吸收的部分

根据 Typeless 中文官网，其语音输入体验值得借鉴的部分包括：

- 跨应用工作，用户不需要切换到专门编辑器。
- 口述结束并完成格式化后，才把最终结果插入目标文本框。
- 去除明显填充词、重复和口头自我修正。
- 自动处理标点、列表和基础文本格式。
- 支持个人词典、人名、术语和表达。
- 支持多语言和中途切换语言。
- 强调零云端数据保留、不使用听写数据训练、历史保存在设备端。

Saymore
吸收的是自然、准确、低延迟的输入体验，不吸收问答、面向任意文本的通用改写或搜索能力。

### 3.2 闪电说值得吸收的部分

根据闪电说官网，“直接说”模式具备以下参考价值：

- 通过全局快捷键在微信、飞书、浏览器、邮箱和终端等应用输入。
- 识别结果直接进入当前输入框。
- 通过词典提高人名、产品名和专业术语的准确率。
- 强调快速反馈和本地处理。
- Mac 与 Windows 已提供下载，Android 与 Linux 处于内测，iOS 标注即将发布。

闪电说的“帮我说”、沟通记忆、知识库和技能不属于 Saymore 当前范围。

### 3.3 Saymore 明确不做的能力

- 语音问答和聊天机器人。
- 根据屏幕内容生成回复。
- 选中文本后的摘要、解释、翻译或改写。
- 联系人记忆、沟通画像和聊天历史读取。
- 知识库、RAG、工具调用和技能执行。
- 自动发送消息。
- 通过 LLM 扩写、续写、补充事实或回答问题。

这些不是“以后顺便加”的隐藏路线，而是当前产品边界之外的独立产品方向。

## 4. 产品主张

Saymore 的差异化不是“免费 Typeless”，而是以下四点的组合：

1. **Local-first**：默认提供可以离线完成的语音识别路径。
2. **Provider-agnostic**：ASR 和 LLM 都有 Provider 边界，不绑定单一供应商。
3. **Transparent**：用户能看到识别和精炼状态、使用的模型、数据是否离开设备。
4. **Reliable**：识别失败、网络失败或投递失败时，用户说过的内容不会静默丢失。

## 5. 目标用户与英雄路径

### 5.1 首要用户

首发面向每天在电脑上高频输入文字的人：

- 开发者：写 Prompt、Issue、代码评审和技术文档。
- 产品与运营：写需求、邮件、社交内容和客户消息。
- 独立开发者与社区贡献者：重视隐私、可配置性和模型选择权。
- 中文为主、经常中英混说的知识工作者。

### 5.2 第一条英雄路径

```text
用户把光标放进任意输入框
-> 按全局快捷键
-> 说一段包含停顿、自我修正和中英混合的话
-> 口述和处理期间不向任何输入框写入内容
-> 松开或再次按键结束
-> Saymore 完成识别、安全本地清理和可选 LLM 精炼
-> 最终结果一次性写入投递时光标所在位置
-> 用户继续编辑或发送
```

“每天愿意一直开着”取决于准确率、延迟、资源占用和投递可靠性，而不是附加 AI
功能数量。

## 6. 产品功能

### 6.1 实时语音识别

- 麦克风默认使用自动检测，用户可以手动固定输入设备。
- 自动检测模式在每次录音开始时读取 macOS
  当前默认输入设备；系统默认设备变化只影响下一次录音，Saymore
  不维护额外的隐藏设备优先级。
- 固定设备只能由用户明确选择；固定设备不可用时明确提示，不静默切换。
- 录音过程中麦克风断开或失效时，立即结束录音并恢复系统音量，保留已经采集的音频继续识别，同时提示“麦克风已断开”。
- 同一次录音中途不自动切换麦克风，避免设备格式、声道和时间戳变化造成丢字。
- 默认使用切换式口述：完整单击一次全局快捷键开始录音，再完整单击一次结束并进入识别，`Esc`
  取消本次口述。
- macOS 当前默认全局快捷键为右 `Command`：第一次单击开始，第二次单击结束。
- 单独修饰键快捷键必须将按下和松开事件成对透传或成对拦截，不能给其他应用留下卡住的修饰键状态。
- 首次引导验证当前键盘和系统能否可靠捕获右
  `Command`；捕获失败或存在冲突时，要求用户设置自定义快捷键。
- 首次安装只创建一个主触发器；设置页提供“添加触发方式”，用户可额外添加键盘组合键或鼠标按键。
- 用户点击主触发器输入框即可直接录入并替换主触发器；“添加另一个”只用于增加并行的附加触发器。
- 每个触发器均可单独编辑或删除，包括最后一个触发器。没有可用触发器时不自动补回默认值，首页以红色“未启用”状态提醒用户进入设置；重新添加后恢复“已启用”。
- 自定义键盘快捷键除默认的单独右 `Command`
  外，必须包含至少一个修饰键和一个普通键；禁止单个字母或数字，并在保存前检测系统与
  Saymore 内部冲突。
- 自定义触发器也可绑定鼠标中键或侧键等非主按键，第一次点击开始、第二次点击结束；禁止绑定左键、右键或滚轮滚动。
- 鼠标绑定只有在系统能可靠捕获按下与释放事件时才能保存，并明确提示所需的 macOS
  辅助功能权限。
- 已绑定的鼠标按键事件由 Saymore
  消费，不再传给前台应用；绑定界面必须明确提示原有的后退、前进或中键行为将被替代。
- 只有 Saymore
  处于可录音状态时才消费已绑定鼠标事件；应用暂停、退出，或权限、麦克风、模型不可用时，事件恢复传给前台应用。
- 键盘和鼠标绑定可以同时存在，任一触发器都进入同一个录音状态机。
- 多个触发器并存时，第一个触发器启动录音；录音期间再次触发任一已绑定触发器都会结束同一会话。
- 菜单栏提供“暂停
  Saymore”；暂停时应用和设置入口仍可用，但禁用全部录音触发器，已绑定鼠标事件恢复透传。
- 暂停状态持久化；应用重启或登录时启动后仍保持暂停，只有用户从菜单栏明确恢复才重新启用全局触发器。
- 录音期间执行暂停等同于取消当前口述：立即停止录音和 ASR，本次音频只在内存中保留
  2 秒用于撤销；未撤销则到期销毁，不投递、不写历史，并恢复被静音的系统音频。
- 录音期间执行“退出 Saymore”或 `Cmd+Q` 时，取消当前口述并立即退出。
- 已进入 ASR、规则或 LLM
  处理时执行退出，显示“等待完成后退出”和“立即退出并丢弃”；等待选项必须有处理超时，不能无限阻塞退出。
- 选择等待后不再尝试直接投递：本地历史开启时，处理完成后把最终结果保存到历史再退出；历史关闭时显示结果浮层，用户复制或明确放弃后才退出。
- 等待处理完成最多持续 30
  秒；超时后只提供“继续等待”和“立即退出并丢弃”，不能自动丢弃。
- 用户可在设置中改为点击模式：按一下开始，再按一下结束。
- 取消会立即终止当前阶段，并提供 2
  秒撤销窗口。窗口内音频只保留在内存，撤销后重新走同一识别、精炼和投递链路；未撤销则立即销毁，不投递也不写入历史。
- 口述和处理期间不向目标输入框写入部分结果。
- 状态浮条默认显示，只展示录音、处理中、成功/失败和音量反馈，不展示部分转录文本。
- 用户可关闭状态浮条，关闭后使用提示音与菜单栏图标反馈状态。
- 录音开始和成功投递结束的交互声音默认开启，用户可关闭；主动取消和失败只使用视觉反馈。
- “语音输入时静音其他活动音频”默认开启，用户可关闭；结束或取消后必须恢复原有音频状态。
- 静音开始前记录当前系统输出音量；结束、取消、失败或应用退出时恢复该值。
- 如果用户在录音期间主动调整系统音量，则该新设置归用户所有，Saymore
  不再恢复录音前的旧值。
- 开始提示音先播放，再执行静音；成功投递结束时先恢复音量，再播放结束提示音；取消或失败时恢复音量但不播放提示音。
- 成功投递且结束提示音开启时，状态浮条保留当前处理状态直到提示音开始播放，并同步执行退出动画；关闭提示音时完成后立即退出。
- 用户结束录音后，处理指示器立即开始并在转写、润色和投递期间连续运动；文本投递不得阻塞动画，结束提示音触发退出时才停止。
- 静音只持续到麦克风实际录音结束；用户释放触发器后立即恢复，ASR、安全本地清理和
  LLM 精炼期间不继续静音。
- 投递失败时的结果浮层独立于状态浮条设置，始终显示。
- 单次口述最长 10 分钟；9 分钟时显示不打断录音的信息面板，提示剩余 60
  秒，用户可确认关闭；达到上限后自动结束并继续处理。
- Provider 限制短于 10 分钟时，界面显示本次实际可用上限。
- 支持连续口述、自然停顿和中英混说。
- 允许用户选择本地或云端 ASR Provider。
- Paraformer、Whisper large-v3-turbo、Qwen3-ASR 1.7B
  和云端识别服务都由用户主动选择，不设置默认推荐项。未来 Saymore 自有云端 ASR
  服务作为新的可选 Provider 加入同一选择体系，不自动取代用户当前选择。
- 尚未配置可用 ASR Provider 时，快捷键不开始录音；Saymore
  先显示原因和“去配置”入口，只有用户确认后才打开模型配置页。
- 模型页中的 API Key、服务地址和模型名先作为未保存草稿处理；编辑、删除或按 Enter
  都不覆盖当前配置。用户点击“保存配置”后先执行连接测试，只有测试成功才原子替换
  当前 Provider 配置；测试失败时保留旧配置并明确显示“连接异常”。
- 识别服务失败时保留已经得到的部分结果。
- VAD 未检测到有效语音时，不调用 ASR、安全本地清理或
  LLM，不投递、不写历史，也不打开结果浮层；只播放轻量提示并短暂显示“未检测到语音”。
- 无有效语音属于正常的空结果，不记为用户取消或识别故障。

### 6.2 多语言与语言策略

- 每个 ASR Provider
  声明自己的语言、自动检测、句内混说、地区变体、热词、流式和离线能力。
- 产品不维护脱离 Provider 的全局语言列表，也不把模型宣传能力直接当作正式支持。
- 用户可选自动检测、偏好语言或固定语言；只显示当前 Provider 能兑现的策略。
- 默认采用
  `Preferred(系统语言)`：仍允许识别其他语言，但在短句和歧义场景优先系统语言。
- Provider 不支持偏好语言时，自动降级为它支持的 `AutoDetect` 或 `Fixed`。
- LLM 精炼接收 ASR 已判定的语言和用户文字偏好，但不承担语言识别，也不能伪装 ASR
  不支持的语种。
- 正式支持范围按 Provider 和模型版本建立验收矩阵，未通过组合标记为实验性。

### 6.3 个人词典与热词

- 个人词典包含用户在应用内手动添加、通过 CSV
  导入，或由高可信用户纠正自动加入的正式词条；自动加入是当前启用的核心能力。
- 当前词典条目包含标准写法、可选语言标签、来源和时间；适用于人名、产品名、缩写和专业术语。旧的常见错误写法字段和相关表为兼容已有数据及未来方案保留，但当前
  UI、CSV 和运行链路不创建或消费语义映射。
- MVP 词典条目全局生效并按语言隔离，不读取当前应用身份，也不支持应用级作用域。
- 已废弃的“比较 ASR 与 LLM
  或最终文本差异并生成语义映射”机制继续禁用；当前学习信号是用户对 Saymore
  刚投递到同一控件、同一局部范围的稳定纠正。
- `term_observations`、`dictionary_candidates` 和 `dictionary_suppressions`
  是当前自动学习数据；只保存最终候选、次数、时间、决策和必要元数据，不保存完整输入框或整段上下文。旧错误写法表只作兼容数据，不参与全局替换。
- 自动加入采用置信度分层：高置信候选累计两次独立纠正后加入，中置信候选继续累计更多独立纠正证据，低置信的一次性句子改写拒绝。中文不统一要求确认，英文品牌、缩写、大小写敏感拼写、代码标识符和中英文行业术语均可自动加入。
- 判断为接受且置信度至少 80% 的候选，在 30 天内累计至少 2 次纠正、覆盖至少 2
  次独立听写后加入；中置信候选需要至少 5 次纠正、覆盖至少 3
  次独立听写。重复次数不能单独证明候选是词条，仍由结构硬门槛、候选类型和可选语义分类共同决定晋升。
- macOS 通过 Accessibility
  只观察刚投递的局部范围和有限时间窗；密码框、安全输入、未知敏感控件和不可读取范围明确跳过，不扫描屏幕、联系人、知识库或其他输入框。
- 词典候选分类使用独立于 Prompt V2
  的版本化结构化提示词，一次完成句法结构、候选提取、词性/类型和专业术语判断，不串联四次模型调用。LLM
  是重要的语义信号但不是唯一入口；未配置 LLM
  时，高词形置信度或更多独立纠正仍可完成自动晋升。
- 普通用户只看到已加入的自动词条、撤销和“不再学习”；候选次数、独立口述数、时间、分数、过滤原因和模型决策只在
  Development 诊断中展示。
- Provider 支持热词时直接传入识别引擎。
- LLM
  只接收标准写法已经在本次转录中以大小写或全角/半角等价形式出现的相关词条，不接收旧错误写法映射。
- LLM 之后执行保守的本地标准写法规范化，例如 `openai`、`OPENAI`、`ＯｐｅｎＡＩ`
  统一为 `OpenAI`；`open ai` 不会被拼接为 `OpenAI`。
- 标准写法规范化只匹配完整 ASCII 字母数字 token，并避开
  URL、邮箱、路径、域名和下划线标识符；禁止任意子字符串替换和语义猜测。
- LLM 只接收本次转录相关的已确认标准词，不接收完整个人词典。
- 词典页面提供自动/手动来源筛选、手动添加、删除和 CSV
  导入；不向普通用户暴露候选审核队列、Markdown 文件或导出。
- CSV
  用于从其他产品批量导入，最多两列：标准写法、可选语言标签。导入项视为用户手动添加；重复项跳过且不能静默覆盖已有标准写法，旧的第三列错误写法格式明确拒绝。

### 6.4 LLM 文本精炼

这里的 LLM
指能处理多种自然语言任务的大语言模型。模型本身通常具备问答、写作和推理能力，但
Saymore 只开放“转录精炼”这一项受约束能力，不提供聊天入口。

Saymore 按模型费用与服务责任划分两种 LLM 使用方式：

- **当前，用户自带 API（BYOK）**：用户使用自己的 Provider
  账户并承担模型费用。产品 UI
  同时提供内置厂商预设和自定义兼容接口；自定义接口可配置 Base URL、Model 和可选
  API Key，也可连接 Ollama、LM Studio 等本地服务。预设和自定义兼容接口都属于
  BYOK，二者只是配置体验不同，不是免费与付费的功能边界。
- **未来，Saymore 托管服务**：用户通过 Saymore 账号和套餐使用由 Saymore
  维护的模型额度与服务，不需要配置 Provider API Key、Base URL 或
  Model。它作为另一个 `LlmProvider`
  实现接入，复用同一精炼请求、响应、超时和降级契约。

当前已适配 SenseNova 与 DeepSeek 两个内置 LLM Provider 预设，以及用户可配置的
OpenAI-compatible 自定义 Provider。自定义配置允许用户提供 Base URL、Model、API
Key 和可选请求头，也可连接 Ollama、LM Studio 等本地服务；原生 Anthropic、Gemini
等非兼容协议仍作为后续独立 Provider 实现。

LLM 精炼的开关状态与 Provider 可用性分开管理：

- 首次安装时 LLM 精炼保持关闭，Provider 状态为“待配置”；首次引导不展示或配置
  LLM。
- 用户完成首次引导后，可在模型页选择内置预设或自定义 Provider；云端 Provider
  必须先完成数据发送确认和连接测试。
- LLM Provider 的配置同样遵循“草稿、连接测试、成功后保存”的顺序；未点击保存或
  连接失败都不改变当前生效配置。
- Provider 未配置或连接测试未通过时，系统自动旁路 LLM 精炼，不阻塞语音输入。
- Provider 配置并测试成功后，后续口述自动进入 LLM 精炼。
- 连接测试只验证服务地址、认证、模型调用和响应协议，不用单条合成样例评判模型的精炼质量。

LLM 可以：

- 自动标点和分段。
- 数字、日期、时间、金额和常见单位格式化。
- 英文大小写和常见专有名词修正。
- 删除填充词、口语重复和 ASR 产生的连续重复片段。
- 处理用户明确表达的口头修正，例如“周三，不对，周四”。
- 把口述的列表、步骤和段落整理为清晰结构。
- 在不改变正式程度、情绪和立场的前提下修正病句和轻微调整语序；当前版本遵循“能删、能调序，但尽量不换词”。
- 根据语义变化分段，不按固定字数拆段；短口述不为了排版强制分段。
- 只在用户明显表达并列项目或有序步骤时输出纯文本项目符号或编号，不生成 Markdown
  标题、引用、任务框、代码围栏、HTML 或 RTF。
- 在用户已经口述称呼、正文或署名时保守整理邮件与正式消息结构，不补写缺失内容。
- 结合上下文删除无意义的犹豫音、卡顿和重新开始，保留承担强调、情绪、顺序或指代作用的重复与口语词。
- 本次相关的已确认标准词可作为拼写提示，但当前不提供错误写法到标准词的语义纠错授权；LLM
  后仍只执行大小写和全角/半角等价的本地格式规范化。

LLM 不可以：

- 补充用户没有说出的事实。
- 改变语气、观点或表达目的。
- 总结、扩写、翻译或回答问题。
- 根据应用内容推测用户想说什么。
- 把模糊表达擅自具体化，或把随意表达统一改成正式公文语气。
- 改写网址、邮箱、文件路径、命令、版本号或代码标识符等受保护内容。

当前版本不提供精炼强度档位。未来可以评估允许更明显改写措辞的“最大程度润色”，但它不属于首版，不能提前放宽当前保守模式的约束。

LLM
未配置、被关闭或不可用时，系统跳过智能精炼并投递安全本地清理结果。每次口述只在
ASR 产生最终文本后发起一次完整、非流式 LLM 请求，不流式发送 ASR
部分结果，也不把长文本拆成多个精炼请求。Chat Completions
请求显式关闭模型推理，使用 `0.2` 的采样温度，并按“原文字数 × 1.5 +
32”设置最大输出 Token，结果限制在 32 到 8192 Token
之间，避免供应商默认推理和过大的默认输出上限增加延迟。单次请求最多等待 8
秒，失败后不自动重试；任何 LLM 故障都不能阻断文字投递。

传输、超时等临时故障连续发生 3 次后，系统暂停调用该 Provider 5
分钟，期间直接投递安全本地清理结果，之后自动尝试恢复；认证、模型或配置错误立即把
Provider
标记为不可用，直到用户修改配置并重新测试。精炼未完成时仍正常投递，并在状态浮条短暂显示“润色未完成，已使用转写结果”。

状态浮条使用用户可理解的处理文案，不显示内部技术术语。停止录音后先显示“正在转写”；只有通过短文本、配置和熔断检查并真正开始
Provider 请求时，才切换为“正在润色”。大模型关闭、短文本正常跳过、Provider
未配置或暂停时始终保留“正在转写”；调用失败后再按原因显示降级结果。代码和技术文档仍可使用“精炼”描述内部领域能力。

LLM
返回后必须执行确定性输出检查。空输出、异常膨胀、受保护内容丢失或改坏，以及包含回答、解释或其他非精炼包装内容的结果均被拒绝；拒绝后投递安全本地清理结果。输出检查必须理解本次传入的已确认标准词，只放行边界完整、大小写或全角/半角等价的标准写法变化，不能把
PostgreSQL、SenseNova、DeepSeek、GitHub
等正确格式修正误判为篡改；URL、邮箱、路径、版本、命令、子字符串误替换和任何语义别名变化仍须拒绝。具体长度阈值通过真实精炼测试集校准，不在产品设计阶段任意指定。

### 6.5 跨应用文本投递

- Secure 输入控件不阻止用户开始口述。投递时若目标为 Secure 控件，Saymore
  只使用短暂的受限剪贴板粘贴尝试，不读回、不验证安全控件内容。
- 口述结束并完成所有已启用阶段后，一次性把最终结果写入投递时光标所在的可编辑控件。
- 最终文本尽量作为目标应用中的一次原子编辑写入，使原生 `Cmd+Z` 可以整段撤销。
- MVP 不提供 Saymore 全局撤销，也不模拟
  `Cmd+Z`；不支持原生撤销的控件不承诺自动回滚。
- 投递目标在实际投递时解析，不绑定口述开始时的应用或输入框。
- 投递时存在文本选区则用最终文本一次性替换该选区；只有光标时在当前光标位置插入。
- Saymore 不读取选区文本，也不把选区内容发送给 ASR、安全本地清理或
  LLM；选区同样以实际投递时状态为准。
- 找不到可编辑光标或直接写入失败时，显示包含最终文本的结果浮层。
- 辅助功能权限未授予或被撤销时进入“仅结果浮层模式”：仍允许录音、识别和精炼，但不尝试直接写入目标控件，最终文本始终进入结果浮层。
- 仅结果浮层模式持续提供前往系统设置授权的入口；授权恢复后，下一次口述重新启用直接投递。
- 结果浮层提供复制操作，但不自动覆盖系统剪贴板。直接投递在 AX
  明确失败或目标控件无法验证时，可以临时写入纯文本、触发粘贴并恢复用户原剪贴板；恢复快照只存在于单次操作内存，不进入本地历史。
- 结果浮层和历史只复制纯文本，保留换行、列表符号与 Markdown 字符，不写入 HTML 或
  RTF 剪贴板格式。
- 直接投递使用同一份纯文本结果，避免目标应用之间出现富文本格式差异。
- 复制成功后自动关闭；用户也可主动关闭。
- 未复制、未关闭时保持显示，不因失焦消失。
- 开始下一次口述时，若本地历史已启用，旧结果先保存到历史，再关闭旧浮层。
- 结果浮层中的文本只读，不承担编辑器职责。
- 投递时焦点已经切换到 Secure
  输入控件，则执行受限粘贴尝试并立即恢复用户剪贴板。无论尝试成功与否，都不显示结果浮层、不保存历史、不保留剪贴板副本；尝试失败后清除本次会话数据。

### 6.6 本地历史

- 普通发布只保存每次口述的最终输出，不保存音频、ASR 原始转录或 LLM 中间响应。
- 调试构建或显式启用 `history-experiments` 的测试构建可在同一加密 payload
  中额外保存 ASR 原始结果和成功通过输出检查的 LLM 精炼结果；历史详情按
  ASR、LLM、最终交付三阶段显示，未调用或未成功精炼时省略 LLM 阶段。
- 每条记录包含最终文本、时间、投递是否成功、实际启用的处理阶段和精炼状态。精炼状态区分“精炼完成”“未使用精炼”“精炼超时”“Provider
  不可用”和“精炼结果被拒绝”；不保存 Provider 原始响应。
- 历史页支持查看、复制、删除单条和清空全部。
- 历史列表把最终文本中的连续空白折叠为空格后单行省略预览，避免列表、段落等格式撑开固定行高；详情和复制仍使用保留原始换行与符号的完整纯文本。
- 本地历史默认开启，默认保留 7 天。
- 用户可以选择关闭、1 天、7 天、30 天或永久保留。
- 用户缩短保留期限时，立即删除已经超出新期限的记录。
- 历史关闭时，主动关闭未复制的结果浮层会放弃该内容。
- MVP 不提供历史全文搜索、标签、收藏或云同步。

## 7. MVP 边界

### 7.1 必须有

- macOS 与 Windows 桌面端的主要听写流程和用户功能保持基本同步；macOS Apple
  Silicon 仍是本地模型性能的首要验收基线。
- 任意可编辑输入框中的全局快捷键输入。
- macOS 默认单击右 `Command` 开始或结束录音，并支持自定义快捷键。
- 自定义触发器支持受约束的键盘组合键，以及鼠标中键或侧键。
- 模型页“语音模型”分类下的 Paraformer、Whisper large-v3-turbo 和 Qwen3-ASR 1.7B
  三个本地 ASR 模型与 Provider，但不默认下载、安装或启用。三个模型都必须在 macOS
  与 Windows
  完成安装、加载和真实听写闭环后才能发布第一版，不用“即将推出”占位代替适配。
- 首次引导不要求配置 ASR
  API、不下载模型，也不发起真实识别；完成引导后，用户自行进入独立的“模型”页，确认下载本地语音模型，或配置
  BYOK 云端 ASR API。模型不打包进应用安装程序。
- 流式部分识别和最终识别。
- ASR Provider 能力声明与自动检测、偏好语言、固定语言三种策略。
- 默认语言策略为 `Preferred(系统语言)`，并按 Provider 能力降级。
- 结束口述后一次性写入最终结果，目标输入框不展示部分识别。
- 默认显示不含转录文本的状态浮条，并允许用户关闭。
- 单次口述最长 10 分钟，达到上限后保留已有内容并自动进入处理。
- 安全本地清理始终开启；LLM
  精炼在首次云端数据发送确认后开启并可独立关闭，产品不提供安全本地清理开关。
- 支持用户在界面中配置自定义 LLM Provider，包括 Base URL、Model、可选 API
  Key、模型发现、连接测试与数据发送确认。
- 当前版本已提供 SenseNova 与 DeepSeek 两个内置预设、OpenAI-compatible 自定义
  Provider、本地无密钥服务、自定义请求头和一个大模型润色开关。
- 明确显示 LLM Provider 的待配置、可用和不可用状态。
- 中英混说、手动词典和 CSV 导入、基于用户局部纠正的自动词典学习，以及 LLM
  后的保守标准写法规范化。ASR/LLM 输出差异不作为自动学习信号。
- 达到精炼阈值的 LLM
  文本支持基础标点、数字格式、自动语义分段、纯文本列表和编号步骤；首版不承诺把口述的“换行”或“换一段”解释为排版命令。
- 可关闭的本地识别历史。
- 历史记录的查看、复制、单条删除、全部清空和按期限自动清理。
- “登录时启动”指用户登录操作系统后自动启动
  Saymore；该选项默认关闭，只在用户明确开启后注册，并允许随时关闭后继续手动启动。
- Dock 图标默认显示，并允许用户关闭；菜单栏入口始终保留。
- 关闭设置或历史窗口只关闭窗口，Saymore 继续在菜单栏常驻；只有菜单栏“退出
  Saymore”或 `Cmd+Q` 才终止进程。
- 麦克风自动检测、交互声音和语音输入时静音其他音频默认开启，并分别允许关闭。
- 音频设置中，“识别提示音”和“识别时静音系统声音”在桌面宽度下同一行左右分栏展示，窄屏下按相同顺序纵向堆叠。
- 麦克风、辅助功能和网络权限提示。
- 投递失败时显示可复制最终文本的结果浮层。
- Secure
  输入控件允许口述，但投递只允许短暂的受限粘贴尝试；转录、剪贴板快照和安全投递结果不能进入结果浮层与历史路径。
- 辅助功能权限缺失时仍可使用仅结果浮层模式。
- 成功投递尽量形成一个原生撤销步骤，但不提供 Saymore 全局撤销。

### 7.2 首次权限引导

首次引导的详细产品合同以 [Saymore Onboarding V1](onboarding-v1.md)
为准。主路线只保留以下边界：

- 引导固定为欢迎、麦克风与快捷键、辅助功能、完成四步，并允许因硬件或权限问题暂时跳过。
- 用户完成或跳过引导后，如果 macOS
  辅助功能权限仍未开启，首次进入主窗口时主动显示一次
  权限提醒；默认的单修饰键全局快捷键在未授权时无法可靠观测，因此不能等待用户按快捷键后
  才提供反馈。
- 麦克风步骤先解释用途再申请权限；授权后只运行独立的本地电平测试，不保存音频，不调用
  ASR 或 LLM。
- 快捷键测试只验证触发器能否可靠捕获并驱动本地麦克风电平会话；录入、冲突校验、持久化和运行时更新与设置页共用同一业务模块。
- 辅助功能步骤解释全局快捷键和跨应用投递用途，持续读取系统当前授权状态，并提供“打开系统设置”和“重新检测”。
- 引导不要求账号、不配置 ASR API 或 LLM
  Provider、不下载模型，也不执行第一次真实听写或外部文本投递。
- 完成或跳过后进入主应用；没有可用 ASR Provider
  时禁用真实听写，并在首页和模型页提供明确的配置入口。
- 完成页提供“登录时启动
  Saymore”开关，默认关闭，只有用户明确开启后才注册系统登录项。
- `completed` 和 `skipped`
  都阻止以后自动弹出；设置页保留“重新运行新手引导”，删除全部本地数据后重新出现。
- MVP
  不申请屏幕录制、通讯录或完整磁盘访问权限；权限被拒绝后不循环触发系统授权对话框。

### 7.3 首版明确不做

- 任何问答或语音助手功能。
- 超出转录精炼范围的通用 LLM 问答与内容生成。
- 屏幕、选区或聊天上下文读取。
- 回复生成、知识库、记忆和技能。
- 移动端输入法。
- 自动发送消息。
- 多人云同步。
- 按目标应用区分的词典作用域。
- 自训练基础语音模型。
- 模型市场、在线模型目录或社区模型提交。
- 为增加数量而提供多个用途重叠的本地模型档位。
- 模型安装趋势、社区评分、排行榜或托管 benchmark 服务。

### 7.4 平台状态与顺序

1. macOS：主要听写流程、Accessibility 投递、全局快捷键、菜单栏常驻和正式打包链路
   已实现。Apple Silicon 仍是本地模型性能的首要验收基线；发布流程同时生成 Apple
   Silicon 与 Intel 的通用安装包。
2. Windows：主要听写流程和用户功能已经与 macOS 基本同步，并使用 Windows 原生的
   音频、全局快捷键、UI Automation、凭据、窗口生命周期与安装适配器。后续工作以
   两个平台的细节对齐、兼容性验证和发布质量为主，而不是重新实现一条 Windows 产品
   流程。
3. Linux：分别评估 X11 与 Wayland，不能预先承诺统一能力。
4. iOS/Android：涉及系统输入法扩展和更严格资源限制，另立规划。

Intel Mac 已包含在通用安装包中，但本地模型性能和长期兼容性仍以 Apple Silicon
基线为准；在完成独立基准前，不对 Intel Mac 作出同等级性能承诺。

### 7.5 应用更新（签名发布之后的产品收尾）

Developer ID 签名与 Apple 公证属于 macOS 直接发布流程；以下应用内更新能力仍保留
为正式发布要求，但不在当前功能和交互细节打磨阶段实施。待 macOS 核心体验冻结后，
再完成更新协议和客户端安装流程。

- 默认自动检查经过签名的新版本，但不自动下载或安装。
- 发现更新后显示版本、发布日期和发行说明，由用户明确确认下载和安装。
- 未公证的社区构建也允许用户点击后直接下载并更新，不要求跳转浏览器手动替换应用。
- 更新清单和安装包必须使用独立的 Saymore
  更新签名；客户端内置公钥，签名无效时拒绝安装。发布私钥不得进入仓库或客户端。
- 旧的未签名或未公证构建更新后仍可能触发 Gatekeeper，产品必须在确认更新前如实
  提示；新的 Developer ID 签名与公证构建复用同一更新协议。
- 更新检查请求不携带稳定设备标识、词典、历史、候选或使用数据，也不能绕过遥测开关承担统计用途。

### 7.6 macOS 发布信任

- 面向普通用户的 macOS 直接发布产物必须使用 Developer ID 签名并通过 Apple
  Notarization。
- 发布流程必须依次完成构建、签名、公证、产物验证和上传，不能把只完成部分步骤的
  安装包标记为正式发布产物。
- 如果发布未经签名或公证的社区构建，发布页必须根据该具体产物如实标记，并提供对应的
  Gatekeeper 提示。
- 发布产物继续提供 SHA-256 校验和；未来的应用内更新仍必须使用独立的 Saymore 更新
  签名，不能用 Apple 代码签名替代更新清单验证。

### 7.7 分发渠道

- GitHub Releases 是当前直接下载安装包和预览版本的官方渠道；官网链接到对应的具体
  Release。
- Mac App Store 是计划中的 macOS 正式分发渠道。完成商店要求、提交审核和发布流程
  后，官网同时提供对应的商店入口；在实际上架前不能把它描述为当前可用渠道。
- GitHub 直接发布继续提供 macOS 与 Windows 安装包、应用更新清单、Saymore 更新
  签名、SHA-256 和发行说明。
- macOS 直接下载使用 DMG，Mac App Store 使用独立的商店打包与提交流程；应用内更新
  使用更新框架要求的签名归档格式。
- MVP 不提供 PKG，不引入安装脚本，也不要求管理员权限。
- Homebrew Cask
  等其他第三方渠道等安装包格式和更新协议稳定后再支持，不能在首版形成未经维护的
  发布来源。

## 8. 体验原则

- **低打扰**：常驻菜单栏，默认显示不含转录文本的状态浮条；用户可关闭并改用提示音与菜单栏图标。
- **入口可选**：登录时启动默认关闭并由用户明确开启；Dock
  图标默认显示但可独立关闭，菜单栏作为稳定入口保留。
- **随时可停**：菜单栏可暂停或恢复所有全局录音触发器，不要求退出应用。
- **状态可信**：始终明确当前是录音、识别、精炼还是投递。
- **结果忠实**：系统宁可保留口语，也不能替用户创造内容。
- **内容不丢**：取消以外的失败都应保留已经识别的文字。
- **延迟可感知**：结束口述后明确反馈识别和精炼状态，并尽快一次性投递最终结果。
- **隐私可见**：清楚标明音频和转录文本分别在本地处理还是发送到云端。
- **键盘友好**：开始、结束和取消都可用快捷键完成；内容恢复通过结果浮层和本地历史完成。

## 9. 成功指标

指标先用于工程验收，不作为营销承诺：

- 快捷键到录音状态反馈：p95 小于 100 ms。
- 结束说话到最终文本可投递：本地路径 p95 小于 1.5 s。
- 10 分钟连续使用无丢字、无重复投递、无焦点错投。
- 识别或投递失败时，已有文本恢复率 100%。
- 对中文、中英混合和专业术语测试集持续记录 CER/WER 与热词命中率。
- LLM 精炼测试集持续记录语义保真率、误删率和无依据新增率。
- 断网后本地模式仍能完成“录音到投递”。
- 空闲时 CPU 和内存占用不影响日常前台应用。

## 10. 技术总体方案

### 10.1 架构形态

仓库采用与当前产品复杂度匹配的 Rust 分层：

```text
Desktop Shell
  全局快捷键、托盘、可选状态浮条、结果浮层、设置、权限引导
        |
Application
  领域状态、纯规则、识别与精炼会话编排、超时、取消、降级与用例
        ^ ports
Infrastructure
  音频、VAD、ASR、LLM、存储、系统凭据、OS 文本投递
```

现有仓库已经使用 Rust + Slint 完成 macOS
辅助功能权限、全局快捷键和文本投递原型。此前的 Tauri/React
界面已在行为一致后删除，项目不维护第二套前端。

### 10.2 建议的 workspace 演进

```text
crates/
  app/             领域状态、纯规则、识别用例与端口
  infra/           音频、ASR、LLM、存储与系统适配器
  xtask/           模型资源、打包、许可和体积检查
apps/
  desktop/         Slint UI、桌面入口与平台 wiring
```

不预先拆出空 crate。当音频、识别、投递或存储形成独立且可复用的接口时，再从 `app`
或 `infra` 提取对应 crate。

### 10.3 核心状态机

```text
Idle
 -> Arming
 -> Listening
 -> FinalizingAudio
 -> Transcribing
 -> CleaningTranscript
 -> [Refining]
 -> ReadyToDeliver
 -> Delivering
 -> Completed

任意投递前状态 -> Cancelled（丢弃会话内全部内容）
外部依赖失败 -> RecoverableError -> 已识别文本或本地草稿
```

状态机由 `app` 层拥有。UI 只渲染状态并发送命令，ASR Provider 不得直接操作 UI
或输入框。

### 10.4 关键端口

- `AudioSource`：输出标准化 PCM 帧和音频设备事件。
- `VoiceActivityDetector`：判断语音活动区间，不负责识别内容。
- `SpeechRecognizer`：声明识别能力，接收语言策略和音频流，输出部分与最终转录。
- `TranscriptCleaner`：始终执行不会改变语义的空白和明确 ASR 重复清理。
- `TextRefiner`：在语义保真约束下把最终转录精炼成可投递文本。
- `LlmProvider`：封装一次完整、非流式文本精炼请求、取消和能力信息，不暴露具体供应商或认证方式。
- `StandardSpellingNormalizer`：在可选 LLM
  之后执行已确认标准词的确定性格式规范化，不推断语义别名。
- `TextDeliverer`：验证焦点目标并原子投递文本。
- `HistoryStore`：普通发布只持久化最终输出及其最小元数据；调试实验可在同一加密
  payload 中额外保存 ASR 与成功精炼结果，不持久化音频。
- `DictionaryStore`：管理已确认词典条目的列出、新增/更新和删除，不负责观察或候选判断。
- `DictionaryLearningStore`：记录高可信用户纠正产生的最小观察、聚合候选证据、执行晋升和静默策略，不接收
  ASR/LLM 输出差异。
- `DictionaryCandidateClassifier`：对已通过本地归因与隐私门槛的候选执行一次结构化分类，输出候选范围、类型、词性、复用价值、置信度和拒绝原因；实现可以是本地规则、本地模型或经确认的云端
  LLM。
- `ModelStore`：管理模型包的下载、校验、导入、删除和本地可用状态。
- `ProviderConfigStore`：在当前用户专属的版本化配置文件中管理 ASR 与自定义 LLM
  配置和用户自行提供的密钥；不得记录密钥值。
- `SecretStore`：通过操作系统安全存储管理 Saymore 账户凭据和本地历史数据密钥。

新 trait
要说明职责、流式事件顺序、取消语义和实现方契约。错误类型要区分可重试、可降级、权限缺失和永久失败。

## 11. 技术选型建议

### 11.1 桌面壳

桌面端确定采用 Slint + Rust，不再使用 Tauri、React、TypeScript、Vite、Node.js 或
WebView。设置页、首次引导、历史页、状态浮条和结果浮层共享一套 Slint
组件；业务逻辑、平台集成、异步任务和持久化全部由 Rust 实现。

Slint 的 `.slint` 文件是允许进入仓库的编译期声明式 UI
文件，但不承载业务规则。`app` 不依赖 Slint；桌面入口只负责把 UI
回调转换成用例命令，并把应用状态映射为 UI view model。

Slint 生成原生桌面程序，但控件是跨平台自绘组件，不等同于 AppKit 或 WinUI
原生控件。macOS 与 Windows
必须分别进行视觉、键盘、屏幕阅读器、缩放和窗口行为验收。

UI
事件循环不得执行音频、ASR、网络、模型加载或数据库阻塞工作。后台任务通过带类型的
Rust 消息把最小状态更新送回 UI 线程。

最大风险是系统集成：macOS Accessibility、Secure
Input、焦点切换、沙箱权限和签名发布必须尽早做原型。

打包由 Cargo 与 `xtask`
统一编排，平台工具负责系统要求的签名、公证和安装包创建。macOS 目标产物仍是
`.app` 与 DMG；Windows 安装格式在 Windows 垂直切片中最终确定。完整决策见
`../technology-stack.md` 与 `../adr/0003-use-slint-for-the-desktop-ui.md`。

### 11.2 音频

- 使用 `cpal` 作为跨平台采集入口。
- 内部统一为单声道 16 kHz PCM，避免每个 ASR Provider 自行处理设备格式。
- VAD、重采样、环形缓冲和背压放在独立音频管线中。
- 录音会话必须支持取消；取消后只在 2
  秒撤销窗口内将临时音频保留在内存，窗口结束且未撤销时立即销毁。
- 麦克风热插拔和系统休眠后要能恢复。

### 11.3 ASR

先定义 Provider 契约，再通过基准选择默认实现。候选路径：

- 本地可移植基线：`whisper.cpp` 的 Rust 绑定。
- 中文实时候选：基于 ONNX 的流式模型，例如 Paraformer 或 Zipformer 系列。
- 云端候选：兼容用户自带密钥的流式语音 API。

推荐本地模型不能只按论文准确率决定。必须在目标硬件上比较：

- 中文 CER 和英文 WER。
- 中英切换与专有名词准确率。
- 首字延迟、结束延迟和实时率。
- 内存、CPU、功耗、模型体积和加载时间。
- 流式识别、热词、标点和时间戳能力。
- 模型权重与运行时的开源许可。

MVP 在独立“模型”页的“语音模型”分类中提供 Paraformer、Whisper large-v3-turbo 和
Qwen3-ASR 1.7B 三个本地 ASR 模型。Saymore
直接适配现成的上游官方固定制品；首个本地 ASR
版本不训练、微调或自行量化，也不自建模型 OSS
或镜像。许可证审查和发布验收通过后，由用户从上游官方固定地址明确选择和下载：

- 每个精确运行时制品分别维护 macOS 与 Windows
  的最低和推荐设备档位；门槛必须来自目标设备实测，不能把单一机型或模型宣传规格套用于全部模型。
- 必须通过中文、英语、中英混说和语言策略验收。
- 模型下载、内存、CPU 和功耗适合桌面常驻。
- 30 秒以内口述在结束后满足最终转录延迟目标。
- 模型及运行时许可允许 Saymore 引导下载、适配和供终端用户使用。
- 默认验证模型的权重许可证必须允许个人与商业本地推理，且不得限制其他合法使用场景；配套运行时代码必须采用
  OSI 批准的开源许可证。
- 首批本地模型不能要求注册账号、提交个人信息或额外接受模型专属条款；受门控模型只能作为用户自行配置的扩展，不能进入首批内置选择。
- Paraformer 定位为中文优先、低资源、流式档；首版优先复用已跑通的 sherpa-onnx Q8
  制品。
- Whisper 固定为
  `large-v3-turbo`，定位为成熟的多语言兼容档；生产运行时制品必须另外固定，不把
  PyTorch checkpoint 直接等同于桌面可用制品。
- Qwen3-ASR 固定为 1.7B，定位为多语言、中文方言和高质量档；官方 BF16 与社区 GGUF
  Q8 是不同制品，只有通过 macOS、Windows
  双端运行时和资源验收的精确变体才能进入应用清单。
- 首批模型可靠后，团队只根据真实失败样本决定继续沿用、下架或替换制品；模型优化另立决策，不作为本地
  ASR 上线的前置条件。新增模型必须对应明确的语言、硬件档位或质量需求。
- 其他实验模型只在开发环境评测，不进入面向普通用户的在线目录、社区注册表或排行榜。

语言能力属于 Provider 契约：

- `supported_languages`：经过该 Provider 验证的语言与地区变体。
- `automatic_language_detection`：是否可以自动识别本次口述语言。
- `intra_utterance_code_switching`：是否支持一句话内切换语言。
- `regional_variants`、`hotwords`、`streaming`、`offline`：其他可选能力。
- 产品提供 `AutoDetect`、`Preferred(language, variant)` 和
  `Fixed(language, variant)` 三种策略，并根据能力动态启用。
- 默认选择 `Preferred(系统语言)`；不支持时按 Provider 能力降级。
- LLM 精炼只消费 ASR 判定结果，不修改识别语言声明。

模型与应用分开分发：

- 安装程序不包含模型权重。
- 应用与模型独立版本化；每个 Saymore
  版本通过随签名应用内置的适配清单声明受支持的模型版本和兼容范围。
- 适配清单签名只证明 Saymore 对该映射和测试元数据负责，不表示模型由 Saymore
  创建或拥有。
- 每个 Saymore
  验证模型条目必须展示上游项目、固定版本、原始许可证、官方下载地址、文件哈希、兼容范围和测试结果。
- 客户端信任随签名应用内置的适配清单，并校验下载文件的哈希与大小；验证失败时拒绝安装。
- 模型直接从项目维护者提供的官方仓库、官方模型站点或正式发布页下载。地址必须固定到不可变版本或提交，不使用可能漂移的“最新版”地址。
- Saymore
  首版不重新打包、镜像或托管模型权重；上游文件被撤回、替换或不可用时显示来源不可用，不静默切换到未知镜像。
- 客户端只下载权重、分词器和配置等受支持的数据文件；运行时代码必须随经过审查和签名的应用版本发布，模型制品不得包含可执行安装脚本或动态代码。
- 新安装在模型页平级展示三个本地语音模型及云端 ASR
  Provider，通过能力、体积、硬件要求和隐私边界帮助用户选择；不突出默认推荐项，不自动选中或下载。
- 应用只在本地检测操作系统、处理器架构、物理内存、必要 CPU
  指令集和可用推理加速后端，并与适配清单中该模型经过实测的最低和推荐要求比较；检测结果用于安装资格和风险提示，不作为精确性能预测，也不默认上传。
- 已知硬不兼容或低于会导致加载失败、进程崩溃或明显系统风险的最低要求时禁用安装并说明不满足的项目；达到最低要求但低于推荐要求时允许用户确认后安装，并提示可能出现的延迟、内存或功耗问题。
- 用户可以同时安装 Paraformer、Whisper large-v3-turbo 和 Qwen3-ASR 1.7B
  中的多个模型，但同一时刻只启用一个当前 ASR
  Provider。安装或切换不会自动删除其他模型。
- 模型页分别显示每个模型的下载体积、本地占用和安装状态，并显示全部本地语音模型的总磁盘占用。
- 应用升级时，当前模型仍兼容则继续使用；新增本地模型或 Saymore 自有云端 ASR
  只作为新选项展示，不强制切换。
- 当前模型与新应用不兼容时，升级流程必须先准备兼容模型；准备完成前保留旧应用与旧模型可用。
- 首次引导不下载模型。用户完成引导后进入独立的“模型”页，可确认下载本地语音模型，也可选择配置云端
  ASR API；下载前显示名称、大小、许可、本地处理说明和预计磁盘占用。
- 下载使用持久临时文件，支持网络中断和应用重启后自动断点续传、错误重试和最终哈希校验；首版不要求提供手动“暂停”按钮。
- 续传前必须确认上游仍是清单固定的同一制品；服务器不支持 HTTP
  Range、响应身份变化或临时文件不一致时，明确提示并从头下载，不拼接不同制品的数据。
- 模型下载使用持久先进先出队列，同一时间只下载一个模型，其余显示“排队中”和顺序；用户可以取消下载中或排队中的任务，不并行抢占带宽和临时磁盘。
- 用户主动取消下载时，删除队列记录和该任务的全部临时分片并立即释放磁盘；网络中断或应用退出不视为取消，保留可验证断点供自动恢复。
- 创建下载任务前计算剩余下载、临时文件、解压或最终安装副本和安全余量组成的峰值磁盘需求。空间不足时禁用下载并显示尚缺空间及“管理已安装模型”入口。
- 下载过程中磁盘意外耗尽时停止任务并保留可验证断点；用户释放空间后可以继续，残缺模型不得激活。
- 用户选择的新模型必须先完成文件校验、加载和最小自检，成功后才原子替换当前 ASR
  Provider；失败时当前选择保持不变，并显示重试、修复和手动改选入口。
- 识别会话失败时不自动切换到其他已安装模型或云端 ASR；模型删除始终需要用户确认。
- 用户可以删除当前本地模型，但确认对话框必须明确说明删除后语音输入将不可用。删除成功后当前
  ASR Provider 置空并禁用真实听写，不自动改选其他已安装模型或云端服务。
- 用户可删除模型，也可手动导入与签名适配清单中固定制品哈希一致的模型目录，用于离线安装；其他未知模型不进入普通用户流程，也不得标记为“Saymore
  验证”。
- 没有可用模型时允许进入应用设置，但不能开始本地识别。

### 11.4 LLM 文本精炼

- 优先使用 ASR 自带标点、ITN 和热词能力。
- 安全本地清理先处理空白和明确的 ASR
  重复；不能用正则或词表无条件删除“那个”“然后”“就是”等可能具有语义的词。已确认标准词的确定性格式规范化在可选
  LLM 阶段之后执行。
- 启用 LLM 后仍先判断精炼收益：中文和中日韩混合文本按汉字等 CJK
  字符计数，连续英文数字词按两个中文单位折算，少于 20 个单位时不调用；纯英文少于
  8
  个单词时不调用。短转录直接使用安全本地清理结果，记为正常跳过，不记为失败、不触发熔断，也不显示降级提示。首版不使用录音时长参与该判断，阈值作为内部策略而非用户挡位。
- `TextRefiner` 再调用所选 `LlmProvider`
  去除口语冗余、处理自我修正、保守润色和整理结构。
- 自动语义分段以时间阶段、任务、讨论对象或沟通目的发生明确变化为依据；原因与结果、观点与补充说明保持在同一段，不能仅因句号数量机械分段。首版不对“换行”“换一段”等口述词建立本地命令规则。
- LLM
  只在最后一个语义单元完整时添加句末标点；口述结束在未完成的从句、条件、原因、动作、列举或自我修正中时，保留残缺内容且不补全、不添加句末标点，前面已经完整的句子仍保留标点。低于精炼阈值的短转录不做语义完整性推断，保留
  ASR 自带标点。
- 当前提供 SenseNova 与 DeepSeek 两个 OpenAI-compatible Chat Completions
  预设；预设隐藏默认 `Base URL`，并允许用户选择服务返回的 Model。自定义 Provider
  允许用户配置 `Base URL`、`Model`、`API Key` 和可选请求头。
- `API Key`
  可为空，并支持可选自定义请求头；核心只依赖精炼所需的最小请求、响应、超时、取消和错误契约。
- 本地自定义 LLM 通过同一兼容接口接入，不在核心代码中绑定 Ollama 或 LM Studio
  专用 SDK。
- 原生 Anthropic、Gemini 和其他非兼容接口后续各自实现新的 `LlmProvider`。
- 未来的 Saymore 托管 LLM 作为独立付费 Provider
  实现接入。客户端精炼管线不需要改动；用户不配置底层 Provider
  凭据，登录令牌、额度、上游模型和服务端数据策略由该 Provider 与后端负责。
- 当前不实现托管 Provider，也不为了它预建计费、账户或同步系统；只确保
  `LlmProvider` 不把 API Key 认证写死在应用层。
- 精炼请求只包含安全本地清理后的转录、语言和本次相关的已确认词条，不包含精炼强度、完整个人词典、屏幕、选区、聊天记录或联系人信息。
- ASR 部分结果只保存在当前会话内；`TextRefiner`
  只消费最终转录并发起一次非流式请求，不实现增量输入或长文本分块。
- Chat Completions 请求使用 `reasoning_effort: "none"`、`temperature: 0.2`，并把
  `max_tokens` 按原文字数动态限制在 32 到 8192 之间；接入测试必须确认所选兼容
  Provider 接受这些参数。
- 桌面端复用长生命周期的 Tokio 运行时，并按当前 Provider 配置缓存
  `FinalTextProcessor`；配置变化时下一次口述自动重建
  Provider，无需重启应用，同一配置的连续失败暂停状态可以跨口述保留。每次听写开始处理最终文本时固定一次大模型开关与
  Provider 配置快照，再把 `ProcessedText`
  交给统一投递路径。停止录音后先显示“正在转写”；只有最终转录达到阈值并真正开始
  LLM
  请求时才切换为“正在润色”，短转录不显示润色状态。正常结束和撤销取消后的恢复识别必须经过同一处理链路。
- Provider 必须支持取消和失败降级；单次请求超时为 8
  秒，当次失败不重试，并返回安全本地清理结果。
- 连续 3 次临时失败触发 5
  分钟内存态暂停，成功后清零；认证、模型或配置错误持续旁路，直到配置重新测试成功。
- Provider
  原始响应只在当前口述会话内保留，用于降级和诊断；普通发布不写入历史。调试构建或
  `history-experiments` 只保存成功通过输出检查的精炼文本，不保存完整 Provider
  响应。
- 本地结构化诊断日志只记录 ASR/LLM
  阶段、耗时、完成、短转录跳过、超时、错误分类、输出拒绝和熔断状态；禁止记录音频、转录、模型响应、Provider
  地址、API Key 或自定义请求头。日志文件权限为 `0600`，达到 2 MiB
  时只保留一份轮换文件；常规请求失败不上传到 Sentry 或其他远程崩溃服务。
- 使用内置、固定且不可由用户修改的结构化指令、正反示例和输出约束，禁止回答问题、扩写或添加事实。
- 输出检查至少覆盖空结果、异常膨胀、受保护内容变化和非精炼包装内容；检查已确认词条时，只把边界完整且大小写或全角/半角等价的标准词视为相同，不授权旧错误写法到标准词的语义映射，也不放宽
  URL、邮箱、路径、版本、命令和其他技术内容保护；检查失败与 Provider
  失败使用同一降级路径。

### 11.5 文本投递

macOS 对提供 `AXSelectedTextRange` 的标准可编辑控件优先使用 Accessibility
API，并只用写入前后的选区位置验证结果，不读取控件全文或选区原文。光标按写入文本的
UTF-16 长度移动，或控件保持新文本选中，才记为 AX 已验证写入。

录音状态浮条和投递恢复浮层使用 macOS 非激活窗口样式，显示、更新和隐藏时不能激活
Saymore 主窗口或夺走目标输入框焦点。LLM
降级只在现有状态浮条短暂显示原因分类，不创建新弹窗、不自动打开主窗口；详细原因进入本地诊断日志。

AX 明确失败、选区范围始终未变化，或控件不提供可验证范围时，使用临时剪贴板触发
`Command+V`。能观察到光标变化时记为剪贴板已验证；无法读取范围时只记为已尝试，不能冒充成功。若
AX
后的范围发生无法解释的变化，则不再自动粘贴，避免重复输入。无法找到有效输入位置或确认投递失败时，把最终文本交给结果浮层展示和复制。

### 11.6 存储与隐私

- 普通产品设置、触发器、个人词典、自动学习观察、历史和已安装模型元数据使用同一个
  `saymore.sqlite3` 文件中的独立业务表；Provider
  配置继续使用当前用户专属的版本化 `config.json`。
- SQLite 由一个后台工作线程独占连接，使用有界命令队列；启用
  WAL、外键、`synchronous = FULL`、有限
  `busy_timeout`、`secure_delete`、启动完整性检查和 `PRAGMA user_version`
  前向迁移。
- 应用进程在数据目录持有单实例文件锁；第二个进程拒绝启动，避免出现两个独立
  SQLite worker 和两套配置写入流程。
- 数据库位于 macOS `~/Library/Application Support/Saymore` 或 Windows
  `%LOCALAPPDATA%\Saymore`，不允许自定义位置，也不能放入云同步目录。
- 本地听写历史使用 AES-256-GCM 应用层加密：每条记录使用随机 96-bit
  nonce，UUID、创建时间和 payload 版本作为
  AAD；数据库只明文保存这些索引元数据、加密版本、nonce 与 ciphertext。
- 随机 32-byte 历史数据密钥保存在 macOS Keychain 或 Windows Credential
  Manager；凭据服务不可用时不得静默降级为明文。
- 空历史且密钥缺失时自动生成新密钥；已有历史但密钥缺失、尺寸错误或校验失败时锁定历史，不覆盖旧密钥。用户只能显式删除旧历史并重新初始化，词典和设置仍可使用。
- 个人词典及自动学习观察、候选和静默数据不加密，但只保存在本机应用数据目录，不参与同步、遥测或备份上传；用户启用支持热词的云端
  ASR 后，识别会话可把已确认词典的标准词作为识别参数发送给该
  Provider。当前只有用户主动选择的 CSV 导入文件是外部明文词典文件。
- ASR 与自定义 LLM 的用户自备 API Key 保存在版本化 Provider 配置文件中；macOS
  文件权限必须为 `0600`，不得写入日志或听写数据库。
- Provider
  配置表单存在未保存内容时只显示保存操作；保存成功且表单未再修改后才显示独立的连接测试。ASR
  连接测试成功后必须清除等待测试状态并显示已连接。
- Provider 配置按 ASR/LLM 类型分区，每个实例使用稳定
  UUID，允许同类型多个实例；当前 ASR 和 LLM 分别记录激活 ID，未知 Provider
  类型及其 JSON 配置必须原样保留。
- 旧单实例 `version 1/2` 配置首次读取时原子迁移为多实例 `version 3`；Provider
  类型、服务地址或数据范围变化会使原数据发送确认指纹失效。删除激活 LLM
  可直接关闭精炼；删除激活 ASR 前必须选择替代项或进入未配置状态。
- 本地历史默认开启并保留 7 天；用户可关闭或改为 1 天、30 天或永久保留。
- 缩短保留期限时立即清理已经超期的记录。
- 普通发布的历史只保存最终输出、时间、投递状态、启用阶段和精炼完成或降级状态，不保存音频、原始转录或
  LLM 中间响应。
- 调试构建或显式启用 `history-experiments` 的测试构建可把 ASR
  原始结果和成功通过检查的 LLM 精炼结果作为可选字段写入同一份加密
  payload，用于在历史详情中与最终交付结果做三阶段对照；普通发布构建必须省略这两个字段。
- 当前不根据 ASR 与 LLM
  或最终文本的模型差异生成观察；只根据同一投递控件和局部范围内的高可信用户纠正生成自动学习观察。
- 原始音频首版不落盘。后续保存音频时，音频文件与 SQLite 分开存储并以历史 UUID
  关联；用户可选择外置机械硬盘等音频根目录。迁移音频库必须后台执行、显示进度、校验哈希、成功前保留旧文件并支持中断续传，新音频从迁移开始后写入新目录。
- ASR 与 LLM Provider 在设置中分别标识发送的数据、服务地址和数据边界。
- 每个云端 Provider
  首次启用前必须显示一次数据发送确认，列出发送的数据、目标服务地址，以及是否包含本次相关的已确认词典条目或用于词典候选分类的最小纠正片段；用户确认后才能启用。
- 更换 Provider、Base URL 或候选分类数据范围后必须重新确认；纯本地 Provider
  不需要该确认。未确认候选、累计证据、完整输入框和整段上下文永远不发送给任何
  Provider。
- 匿名使用统计和崩溃报告默认分别关闭，只能由用户主动开启。
- 遥测采用字段白名单；即使用户开启，也禁止上传音频、转录文本、历史、词典、候选、Provider
  地址、API Key、请求内容或响应内容。
- Telemetry 默认关闭；崩溃报告发送前可预览。
- 云端 ASR 只发送音频、识别参数和 Provider 支持时的已确认词典热词；云端 LLM
  只发送转录、本次相关的已确认词条和精炼参数，不发送屏幕或应用内容。

## 13. 验证与测试策略

- 领域与 app 层：状态机、取消、8 秒超时、单次调用、降级和幂等投递测试。
- 取消测试：覆盖录音、ASR、安全本地清理和 LLM 精炼阶段，确认 2
  秒撤销窗口内只在内存保留音频，撤销可恢复标准处理链，超时后不投递、不写历史且清除会话数据。
- Provider contract tests：分别用同一套契约验证 ASR Provider 和 LLM Provider。
- 自动词典学习测试：验证 ASR/LLM
  差异不会生成观察；用户局部纠正可以生成候选；高置信英文品牌、代码标识符和中英文术语可自动晋升；普通谓词短语、整句改写、删除、标点和格式调整不会晋升；删除自动词条后不会重新学习。
- 词典候选分类测试：使用固定夹具验证结构化输出、Prompt 版本、无 LLM
  降级、不同文字系统和置信度边界；禁止在测试中调用真实外部 Provider。
- macOS 纠正观察兼容性测试：覆盖原生文本控件、浏览器/Web、Electron
  和富文本控件，并验证 Secure 输入、不可读取范围、失焦、超时和局部锚点隐私边界。
- 标准写法规范化测试：覆盖大小写、全角/半角等价、完整 ASCII token，以及
  URL、邮箱、路径、域名、下划线标识符、空格分词和子字符串不被改写；存储迁移还要验证
  `open ai` 与 `openai` 保持为两个词条身份。
- 词典文件测试：验证最多两列的 CSV

## 14. 主要风险

### 14.1 跨应用投递会直接影响信任

偶发错投、重复粘贴或覆盖剪贴板会让用户立即卸载。平台原型必须早于大规模模型集成。

### 14.2 本地识别与低延迟、低包体存在冲突

高质量本地模型会增加下载、内存和启动时间。需要模型按需下载、硬件分档和云端可选路径。

### 14.3 中英混说与专业词汇准确率不足

通用公开模型未必适合目标用户。必须建立真实测试集，并把个人词典和热词能力作为核心功能，而不是设置页附属项。

### 14.4 最终光标可能不是用户预期的位置

产品明确以实际投递时的光标位置为准，不绑定口述开始位置。用户在处理期间切换窗口或移动光标会改变投递目标；投递时没有可编辑光标则显示结果浮层，避免内容丢失。

### 14.5 模型许可可能限制分发

核心仓库不能分发许可不清晰的权重。模型清单要记录来源、哈希、许可和再分发条件；不允许商业本地推理、限制合法用途或要求门控访问的模型不能成为默认验证模型。

### 14.6 LLM 精炼可能改变用户原意

必须在当前会话内保留原始转录用于降级、限制输入上下文、使用固定保守提示词并检查输出，针对事实新增、异常膨胀、数字变化、否定词丢失、受保护内容变化和专有名词误改建立回归测试。LLM
失败或结果被拒绝时自动回退到安全本地清理结果，不能成为输入链路的单点故障。

### 14.7 上游模型制品需要持续验证

主动选型和适配会带来许可证复核、上游可用性、版本兼容和安全响应成本。即使使用官方来源，也必须固定版本和哈希，防止上游文件漂移或下载损坏。每个新增模型都必须解决现有模型无法覆盖的明确需求，并承担独立回归与回退责任；模型数量不能替代听写成功率、延迟和用户留存。

## 16. 推荐的下一步

桌面技术栈已经确定为 Slint + Rust。下一步按以下顺序推进：

1. **锁定三个运行时制品**：模型身份固定为 Paraformer、Whisper large-v3-turbo 和
   Qwen3-ASR 1.7B；分别固定 macOS 与 Windows
   共用或对应的上游官方版本、运行时格式、下载地址、大小和哈希。
2. **修正 Paraformer 识别结束语义**：先消除当前 Paraformer
   探针结尾重复和截断，再把同一运行时实现为生产 `StreamingSpeechRecognizer`
   Adapter。
3. **重构语音模型页并完成听写闭环**：在“语音模型”分类展示三个本地模型；从上游固定地址下载，校验大小与哈希，原子激活模型，并接入运行时选择、错误恢复和删除流程。
4. **双端 Beta 验收**：在 macOS 与 Windows
   同时覆盖离线首轮识别、连续听写、取消、无语音、模型损坏和目标应用投递；双端通过后同步发布，不等待模型调优和模型平台能力。
