# Saymore 开源语音识别与输入决策地图

状态：产品方向草案，桌面技术栈已决策
日期：2026-07-13

## 1. 终点

产出一套足以进入 PRD 和技术拆票阶段的方向：明确 Saymore 的目标用户、核心价值、首版边界、语音识别架构、风险验证顺序和后续决策。

当前产品只做语音识别、LLM 文本精炼和跨应用文本输入，不做语音助手、问答、回复生成或任务执行。

## 2. 一句话定位

**Saymore 是一个本地优先、模型可替换、能在任何应用中工作的开源语音输入工具。**

用户按下快捷键自然说话，Saymore 实时识别语音，并将准确、格式清晰的文字写入当前输入框。

产品链路严格限定为：

```text
说话 -> 语音识别 -> 安全本地清理 -> [LLM 文本精炼] -> 文本投递
```

安全本地清理始终执行，只处理首尾空白、异常空格、能够确定由 ASR 技术故障造成的重复片段，以及边界和语义都明确的已确认词典映射。用户只控制是否启用 LLM 文本精炼；关闭、未配置或暂时不可用时，系统在安全本地清理后直接投递。LLM 负责把口语转录保守地整理成忠于原意的书面文本，例如去除无意义填充词和口误重复、处理自我修正、补充标点和整理结构。它不回答问题、不扩写用户意图，也不读取当前应用的屏幕内容。

核心听写可靠后，Saymore 的后续差异化方向是中文优先的 ASR Model Hub：让用户发现、比较并一键安装经过兼容性验证的本地模型，同时保留云端 ASR API。该方向不属于 MVP，不改变首版只验证一个默认本地模型档位的边界。完整方向见 [中文优先 ASR Model Hub](asr-model-hub.md)。

## 3. 外部产品启发

### 3.1 Typeless 值得吸收的部分

根据 Typeless 中文官网，其语音输入体验值得借鉴的部分包括：

- 跨应用工作，用户不需要切换到专门编辑器。
- 口述结束并完成格式化后，才把最终结果插入目标文本框。
- 去除明显填充词、重复和口头自我修正。
- 自动处理标点、列表和基础文本格式。
- 支持个人词典、人名、术语和表达。
- 支持多语言和中途切换语言。
- 强调零云端数据保留、不使用听写数据训练、历史保存在设备端。

Saymore 吸收的是自然、准确、低延迟的输入体验，不吸收问答、面向任意文本的通用改写或搜索能力。

### 3.2 闪电说值得吸收的部分

根据闪电说官网，“直接说”模式具备以下参考价值：

- 通过全局快捷键在微信、飞书、浏览器、邮箱和终端等应用输入。
- 识别结果直接进入当前输入框。
- 通过词典提高人名、产品名和专业术语的准确率。
- 强调快速反馈和本地处理。
- Mac 与 Windows 已提供下载，Android 与 Linux 处于内测，iOS 标注即将发布。

闪电说的“帮我说”、沟通记忆、知识库和技能不属于 Saymore 当前范围。

### 3.3 Saymore 明确不做的能力

- 语音问答和聊天机器人。
- 根据屏幕内容生成回复。
- 选中文本后的摘要、解释、翻译或改写。
- 联系人记忆、沟通画像和聊天历史读取。
- 知识库、RAG、工具调用和技能执行。
- 自动发送消息。
- 通过 LLM 扩写、续写、补充事实或回答问题。

这些不是“以后顺便加”的隐藏路线，而是当前产品边界之外的独立产品方向。

## 4. 产品主张

Saymore 的差异化不是“免费 Typeless”，而是以下四点的组合：

1. **Local-first**：默认提供可以离线完成的语音识别路径。
2. **Provider-agnostic**：ASR 和 LLM 都有 Provider 边界，不绑定单一供应商。
3. **Transparent**：用户能看到识别和精炼状态、使用的模型、数据是否离开设备。
4. **Reliable**：识别失败、网络失败或投递失败时，用户说过的内容不会静默丢失。

## 5. 目标用户与英雄路径

### 5.1 首要用户

首发面向每天在电脑上高频输入文字的人：

- 开发者：写 Prompt、Issue、代码评审和技术文档。
- 产品与运营：写需求、邮件、社交内容和客户消息。
- 独立开发者与开源贡献者：重视隐私、可配置性和模型选择权。
- 中文为主、经常中英混说的知识工作者。

### 5.2 第一条英雄路径

```text
用户把光标放进任意输入框
-> 按全局快捷键
-> 说一段包含停顿、自我修正和中英混合的话
-> 口述和处理期间不向任何输入框写入内容
-> 松开或再次按键结束
-> Saymore 完成识别、安全本地清理和可选 LLM 精炼
-> 最终结果一次性写入投递时光标所在位置
-> 用户继续编辑或发送
```

“每天愿意一直开着”取决于准确率、延迟、资源占用和投递可靠性，而不是附加 AI 功能数量。

## 6. 产品功能

### 6.1 实时语音识别

- 麦克风默认使用自动检测，用户可以手动固定输入设备。
- 自动检测模式在每次录音开始时读取 macOS 当前默认输入设备；系统默认设备变化只影响下一次录音，Saymore 不维护额外的隐藏设备优先级。
- 固定设备只能由用户明确选择；固定设备不可用时明确提示，不静默切换。
- 录音过程中麦克风断开或失效时，立即结束录音并恢复系统音量，保留已经采集的音频继续识别，同时提示“麦克风已断开”。
- 同一次录音中途不自动切换麦克风，避免设备格式、声道和时间戳变化造成丢字。
- 默认使用切换式口述：完整单击一次全局快捷键开始录音，再完整单击一次结束并进入识别，`Esc` 取消本次口述。
- macOS 当前默认全局快捷键为右 `Command`：第一次单击开始，第二次单击结束。
- 首次引导验证当前键盘和系统能否可靠捕获右 `Command`；捕获失败或存在冲突时，要求用户设置自定义快捷键。
- 首次安装只创建一个主触发器；设置页提供“添加触发方式”，用户可额外添加键盘组合键或鼠标按键。
- 用户点击主触发器输入框即可直接录入并替换主触发器；“添加另一个”只用于增加并行的附加触发器。
- 每个附加触发器可单独编辑、禁用或删除；系统必须始终保留至少一个可用触发器。
- 自定义键盘快捷键除默认的单独右 `Command` 外，必须包含至少一个修饰键和一个普通键；禁止单个字母或数字，并在保存前检测系统与 Saymore 内部冲突。
- 自定义触发器也可绑定鼠标中键或侧键等非主按键，第一次点击开始、第二次点击结束；禁止绑定左键、右键或滚轮滚动。
- 鼠标绑定只有在系统能可靠捕获按下与释放事件时才能保存，并明确提示所需的 macOS 辅助功能权限。
- 已绑定的鼠标按键事件由 Saymore 消费，不再传给前台应用；绑定界面必须明确提示原有的后退、前进或中键行为将被替代。
- 只有 Saymore 处于可录音状态时才消费已绑定鼠标事件；应用暂停、退出，或权限、麦克风、模型不可用时，事件恢复传给前台应用。
- 键盘和鼠标绑定可以同时存在，任一触发器都进入同一个录音状态机。
- 多个触发器并存时，第一个触发器启动录音；录音期间再次触发任一已绑定触发器都会结束同一会话。
- 菜单栏提供“暂停 Saymore”；暂停时应用和设置入口仍可用，但禁用全部录音触发器，已绑定鼠标事件恢复透传。
- 暂停状态持久化；应用重启或登录时启动后仍保持暂停，只有用户从菜单栏明确恢复才重新启用全局触发器。
- 录音期间执行暂停等同于取消当前口述：清除本次音频和文本，不投递、不写历史，并恢复被静音的系统音频。
- 录音期间执行“退出 Saymore”或 `Cmd+Q` 时，取消当前口述并立即退出。
- 已进入 ASR、规则或 LLM 处理时执行退出，显示“等待完成后退出”和“立即退出并丢弃”；等待选项必须有处理超时，不能无限阻塞退出。
- 选择等待后不再尝试直接投递：本地历史开启时，处理完成后把最终结果保存到历史再退出；历史关闭时显示结果浮层，用户复制或明确放弃后才退出。
- 等待处理完成最多持续 30 秒；超时后只提供“继续等待”和“立即退出并丢弃”，不能自动丢弃。
- 用户可在设置中改为点击模式：按一下开始，再按一下结束。
- 取消会立即终止当前阶段并丢弃本次口述的音频和所有文本，不投递也不写入历史。
- 口述和处理期间不向目标输入框写入部分结果。
- 状态浮条默认显示，只展示录音、处理中、成功/失败和音量反馈，不展示部分转录文本。
- 用户可关闭状态浮条，关闭后使用提示音与菜单栏图标反馈状态。
- 开始、停止、取消、成功和失败的交互声音默认开启，用户可关闭。
- “语音输入时静音其他活动音频”默认开启，用户可关闭；结束或取消后必须恢复原有音频状态。
- 静音开始前记录当前系统输出音量；结束、取消、失败或应用退出时恢复该值。
- 如果用户在录音期间主动调整系统音量，则该新设置归用户所有，Saymore 不再恢复录音前的旧值。
- 开始提示音先播放，再执行静音；结束、取消或失败时先恢复音量，再播放对应提示音。
- 静音只持续到麦克风实际录音结束；用户释放触发器后立即恢复，ASR、安全本地清理和 LLM 精炼期间不继续静音。
- 投递失败时的结果浮层独立于状态浮条设置，始终显示。
- 单次口述最长 10 分钟；9 分钟时提示剩余 60 秒，达到上限后自动结束并继续处理。
- Provider 限制短于 10 分钟时，界面显示本次实际可用上限。
- 支持连续口述、自然停顿和中英混说。
- 允许用户选择本地或云端 ASR Provider。
- 识别服务失败时保留已经得到的部分结果。
- VAD 未检测到有效语音时，不调用 ASR、安全本地清理或 LLM，不投递、不写历史，也不打开结果浮层；只播放轻量提示并短暂显示“未检测到语音”。
- 无有效语音属于正常的空结果，不记为用户取消或识别故障。

### 6.2 多语言与语言策略

- 每个 ASR Provider 声明自己的语言、自动检测、句内混说、地区变体、热词、流式和离线能力。
- 产品不维护脱离 Provider 的全局语言列表，也不把模型宣传能力直接当作正式支持。
- 用户可选自动检测、偏好语言或固定语言；只显示当前 Provider 能兑现的策略。
- 默认采用 `Preferred(系统语言)`：仍允许识别其他语言，但在短句和歧义场景优先系统语言。
- Provider 不支持偏好语言时，自动降级为它支持的 `AutoDetect` 或 `Fixed`。
- LLM 精炼接收 ASR 已判定的语言和用户文字偏好，但不承担语言识别，也不能伪装 ASR 不支持的语种。
- 正式支持范围按 Provider 和模型版本建立验收矩阵，未通过组合标记为实验性。

### 6.3 个人词典与热词

- 个人词典只包含用户已经确认的词典条目；自动发现的内容先进入词汇候选，不直接入库。
- 词典条目包含标准写法、语言标签，以及零个或多个常见错误写法；适用于人名、产品名、缩写和专业术语。
- MVP 词典条目全局生效并按语言隔离，不读取当前应用身份，也不支持应用级作用域。
- 自动词汇发现默认开启，但候选在用户确认前不得影响 ASR、安全本地清理或 LLM 精炼。
- 用户可以确认或拒绝候选；确认后候选才转为词典条目。
- 自动候选采用非打扰式审核：只在个人词典入口显示待确认数量，由用户进入后集中处理；听写结束后不弹出候选提示。
- 同一标准写法或错误写法映射在 30 天内至少出现 3 次、且来自至少 2 次独立听写时，才生成待确认候选；未达到门槛的观察记录不能参与识别或精炼。
- 用户普通拒绝候选后，90 天内不再推荐同一映射；用户选择“永不推荐”后永久屏蔽，直到在个人词典设置中手动恢复。
- 首版只在当前会话内比较原始 ASR 文本与安全本地清理或 LLM 精炼后的最终文本，从重复出现的局部纠错中发现候选。
- 候选发现持久化时只保存错误写法、建议标准写法、语言、出现次数、最近出现时间和审核状态，不保存产生该候选的完整原始文本或最终文本。
- 未达到候选门槛的观察记录滚动保留 30 天，过期自动删除。
- 候选确认后删除观察计数，只保留正式词典条目；普通拒绝后只保留最小映射和 90 天冷却截止时间；永久屏蔽后只保留最小映射和屏蔽状态。
- 自动发现只使用本次语音输入流程中已经授权处理的信号，不监控目标应用中的后续编辑，也不扫描屏幕、联系人或知识库。
- Provider 支持热词时直接传入识别引擎。
- 安全本地清理根据本次转录筛选精确或近似匹配的已确认词典条目。
- 词典映射必须匹配词条语言，并且只作用于完整词或边界明确的短语；多个条目冲突时优先最长匹配。
- 禁止对任意子字符串执行全局替换；无法可靠判断边界时保持原文，或把相关条目交给受约束的 LLM 精炼。
- Provider 不支持热词时只做受约束的替换，不使用生成模型无依据猜测。
- LLM 只接收本次转录相关的候选词条，不接收完整个人词典。
- 词典条目可以导入、导出、禁用和删除；待确认候选可以查看、拒绝和清理。
- JSON 导入导出完整保留标准写法、常见错误写法、语言和启用状态，用于备份与迁移。
- CSV 导入导出用于表格批量编辑词典条目；JSON 和 CSV 都不包含自动候选、观察计数、拒绝冷却或永久屏蔽记录。
- 导入前显示变更预览，并按“语言 + 标准写法”识别重复条目；重复条目的错误写法取并集，本地已有条目的启用状态不被导入文件覆盖。
- 同一语言下，同一个错误写法指向不同标准写法时标记为冲突，必须由用户选择，不能静默覆盖。

### 6.4 LLM 文本精炼

这里的 LLM 指能处理多种自然语言任务的大语言模型。模型本身通常具备问答、写作和推理能力，但 Saymore 只开放“转录精炼”这一项受约束能力，不提供聊天入口。

Saymore 当前实现一种 LLM 使用方式，并为未来默认服务保留同一扩展点：

- **当前，自定义 LLM**：用户配置兼容接口的 Base URL、API Key 和 Model；可连接云端服务，也可连接 Ollama、LM Studio 等本地服务。
- **未来，默认 LLM**：Saymore 托管服务作为另一个 `LlmProvider` 实现接入，复用同一精炼请求、响应、超时和降级契约。

MVP 只支持最小 OpenAI-compatible Chat Completions 契约，配置包括 `Base URL`、`Model`、可空的 `API Key` 和可选自定义请求头。本地服务可以不配置密钥。原生 Anthropic、Gemini 和未来托管服务通过新的 `LlmProvider` 后续加入。

LLM 精炼的开关状态与 Provider 可用性分开管理：

- 首次安装时 LLM 精炼默认开启，Provider 状态为“待配置”。
- 首次引导提供配置入口，但不强制用户完成。
- Provider 未配置或连接测试未通过时，系统自动旁路 LLM 精炼，不阻塞语音输入。
- Provider 配置并测试成功后，后续口述自动进入 LLM 精炼。
- 连接测试只验证服务地址、认证、模型调用和响应协议，不用单条合成样例评判模型的精炼质量。

LLM 可以：

- 自动标点和分段。
- 数字、日期、时间、金额和常见单位格式化。
- 英文大小写和常见专有名词修正。
- 删除填充词、口语重复和 ASR 产生的连续重复片段。
- 处理用户明确表达的口头修正，例如“周三，不对，周四”。
- 把口述的列表、步骤和段落整理为清晰结构。
- 在不改变正式程度、情绪和立场的前提下修正病句和轻微调整语序；当前版本遵循“能删、能调序，但尽量不换词”。
- 根据语义变化分段，不按固定字数拆段；短口述不为了排版强制分段。
- 只在用户明显表达并列项目或有序步骤时输出纯文本项目符号或编号，不生成 Markdown 标题、引用、任务框、代码围栏、HTML 或 RTF。
- 在用户已经口述称呼、正文或署名时保守整理邮件与正式消息结构，不补写缺失内容。
- 在上下文明确是排版指令时，把“换行”转换为一个换行符，把“换一段”转换为两个换行符并删除指令；讨论“自动换行”等普通语义时保留原词。
- 结合上下文删除无意义的犹豫音、卡顿和重新开始，保留承担强调、情绪、顺序或指代作用的重复与口语词。
- 优先采用本次相关的已确认词典条目纠正 ASR 错词；有歧义的常见词仍结合整句判断，不能执行无条件全局替换。

LLM 不可以：

- 补充用户没有说出的事实。
- 改变语气、观点或表达目的。
- 总结、扩写、翻译或回答问题。
- 根据应用内容推测用户想说什么。
- 把模糊表达擅自具体化，或把随意表达统一改成正式公文语气。
- 改写网址、邮箱、文件路径、命令、版本号或代码标识符等受保护内容。

当前版本不提供精炼强度档位。未来可以评估允许更明显改写措辞的“最大程度润色”，但它不属于首版，不能提前放宽当前保守模式的约束。

LLM 未配置、被关闭或不可用时，系统跳过智能精炼并投递安全本地清理结果。每次口述只在 ASR 产生最终文本后发起一次完整、非流式 LLM 请求，不流式发送 ASR 部分结果，也不把长文本拆成多个精炼请求。单次请求最多等待 8 秒，失败后不自动重试；任何 LLM 故障都不能阻断文字投递。

传输、超时等临时故障连续发生 3 次后，系统暂停调用该 Provider 5 分钟，期间直接投递安全本地清理结果，之后自动尝试恢复；认证、模型或配置错误立即把 Provider 标记为不可用，直到用户修改配置并重新测试。精炼未完成时仍正常投递，并在状态浮条短暂显示“精炼未完成，已使用识别结果”。

LLM 返回后必须执行确定性输出检查。空输出、异常膨胀、受保护内容丢失或改坏，以及包含回答、解释或其他非精炼包装内容的结果均被拒绝；拒绝后投递安全本地清理结果。具体长度阈值通过真实精炼测试集校准，不在产品设计阶段任意指定。

### 6.5 跨应用文本投递

- 录音开始时检测到系统标记的 Secure 输入控件，拒绝启动本次口述并提示“不支持安全输入框”。
- 口述结束并完成所有已启用阶段后，一次性把最终结果写入投递时光标所在的可编辑控件。
- 最终文本尽量作为目标应用中的一次原子编辑写入，使原生 `Cmd+Z` 可以整段撤销。
- MVP 不提供 Saymore 全局撤销，也不模拟 `Cmd+Z`；不支持原生撤销的控件不承诺自动回滚。
- 投递目标在实际投递时解析，不绑定口述开始时的应用或输入框。
- 投递时存在文本选区则用最终文本一次性替换该选区；只有光标时在当前光标位置插入。
- Saymore 不读取选区文本，也不把选区内容发送给 ASR、安全本地清理或 LLM；选区同样以实际投递时状态为准。
- 找不到可编辑光标或直接写入失败时，显示包含最终文本的结果浮层。
- 辅助功能权限未授予或被撤销时进入“仅结果浮层模式”：仍允许录音、识别和精炼，但不尝试直接写入目标控件，最终文本始终进入结果浮层。
- 仅结果浮层模式持续提供前往系统设置授权的入口；授权恢复后，下一次口述重新启用直接投递。
- 结果浮层提供复制操作，但不自动覆盖系统剪贴板。直接投递在 AX 明确失败或目标控件无法验证时，可以临时写入纯文本、触发粘贴并恢复用户原剪贴板；恢复快照只存在于单次操作内存，不进入本地历史。
- 结果浮层和历史只复制纯文本，保留换行、列表符号与 Markdown 字符，不写入 HTML 或 RTF 剪贴板格式。
- 直接投递使用同一份纯文本结果，避免目标应用之间出现富文本格式差异。
- 复制成功后自动关闭；用户也可主动关闭。
- 未复制、未关闭时保持显示，不因失焦消失。
- 开始下一次口述时，若本地历史已启用，旧结果先保存到历史，再关闭旧浮层。
- 结果浮层中的文本只读，不承担编辑器职责。
- 投递时焦点已经切换到 Secure 输入控件，则丢弃最终结果，不写入、不显示结果浮层、不保存历史，并清除本次会话数据。

### 6.6 本地历史

- 只保存每次口述的最终输出，不保存音频、ASR 原始转录或 LLM 中间响应。
- 每条记录包含最终文本、时间、投递是否成功、实际启用的处理阶段和精炼状态。精炼状态区分“精炼完成”“未使用精炼”“精炼超时”“Provider 不可用”和“精炼结果被拒绝”；不保存 Provider 原始响应。
- 历史页支持查看、复制、删除单条和清空全部。
- 本地历史默认开启，默认保留 7 天。
- 用户可以选择关闭、1 天、7 天、30 天或永久保留。
- 用户缩短保留期限时，立即删除已经超出新期限的记录。
- 历史关闭时，主动关闭未复制的结果浮层会放弃该内容。
- MVP 不提供历史全文搜索、标签、收藏或云同步。

## 7. MVP 边界

### 7.1 必须有

- macOS Apple Silicon 首发。
- 任意可编辑输入框中的全局快捷键输入。
- macOS 默认单击右 `Command` 开始或结束录音，并支持自定义快捷键。
- 自定义触发器支持受约束的键盘组合键，以及鼠标中键或侧键。
- 一个经过完整验证的“均衡”默认本地 ASR 模型与 Provider。
- 首次引导中确认并下载默认 ASR 模型，模型不打包进应用安装程序。
- 流式部分识别和最终识别。
- ASR Provider 能力声明与自动检测、偏好语言、固定语言三种策略。
- 默认语言策略为 `Preferred(系统语言)`，并按 Provider 能力降级。
- 结束口述后一次性写入最终结果，目标输入框不展示部分识别。
- 默认显示不含转录文本的状态浮条，并允许用户关闭。
- 单次口述最长 10 分钟，达到上限后保留已有内容并自动进入处理。
- 安全本地清理始终开启；LLM 精炼默认开启并可独立关闭，产品不提供安全本地清理开关。
- 支持用户配置自定义 LLM Provider。
- 自定义 LLM 首版采用最小 OpenAI-compatible Chat Completions 契约，支持本地无密钥服务和自定义请求头。
- 明确显示 LLM Provider 的待配置、可用和不可用状态。
- 中英混说、手动词典映射，以及需要用户确认的自动词汇候选。
- LLM 精炼支持基础标点、数字格式、语义分段、纯文本列表和编号步骤。
- 可关闭的本地识别历史。
- 历史记录的查看、复制、单条删除、全部清空和按期限自动清理。
- 登录时启动默认开启，并允许用户关闭后手动启动。
- Dock 图标默认显示，并允许用户关闭；菜单栏入口始终保留。
- 关闭设置或历史窗口只关闭窗口，Saymore 继续在菜单栏常驻；只有菜单栏“退出 Saymore”或 `Cmd+Q` 才终止进程。
- 麦克风自动检测、交互声音和语音输入时静音其他音频默认开启，并分别允许关闭。
- 麦克风、辅助功能和网络权限提示。
- 投递失败时显示可复制最终文本的结果浮层。
- Secure 输入控件必须拒绝录音或投递，且不能进入结果浮层与历史路径。
- 辅助功能权限缺失时仍可使用仅结果浮层模式。
- 成功投递尽量形成一个原生撤销步骤，但不提供 Saymore 全局撤销。

### 7.2 首次权限引导

- 先解释用途并申请麦克风权限；用户准备测试文本投递时，再解释用途并申请辅助功能权限。
- 麦克风权限通过后，展示推荐默认模型的名称、大小、许可证、本地处理说明和磁盘需求；用户确认后开始下载。
- 模型下载期间继续辅助功能授权和触发器设置，不用下载阻塞整个引导。
- 模型完成适配清单签名、文件哈希和兼容性校验后，才允许进入第一次真实听写测试。
- 第一次真实听写不使用应用内专用测试框，也不自动打开或控制 TextEdit；引导用户把光标放进任意外部可编辑输入框，再通过主触发器运行完整的录音、ASR、可选变换和最终投递管线。
- 首次外部投递失败时沿用结果浮层保护文本，不建立只用于 onboarding 的特殊投递路径。
- 第一次听写不要求成功才能结束引导；用户可以跳过进入应用，菜单栏持续提供“完成设置”入口。
- 麦克风、模型、辅助功能或输入设备缺失时，只把依赖该条件的能力标记为不可用并提供修复入口，不能把用户锁在引导页。
- 鼠标触发器复用辅助功能权限，不为它申请额外系统权限。
- MVP 不申请屏幕录制、通讯录或完整磁盘访问权限。
- 权限被拒绝后停留在对应引导步骤，提供“打开系统设置”和“重新检测”；不循环触发系统授权对话框。

### 7.3 首版明确不做

- 任何问答或语音助手功能。
- 超出转录精炼范围的通用 LLM 问答与内容生成。
- 屏幕、选区或聊天上下文读取。
- 回复生成、知识库、记忆和技能。
- 移动端输入法。
- 自动发送消息。
- 多人云同步。
- 按目标应用区分的词典作用域。
- 自训练基础语音模型。
- ASR Model Hub、在线模型目录或社区模型提交。
- 多个正式支持的本地模型档位与面向普通用户的模型市场。
- 模型安装趋势、社区评分、每日排名或托管 benchmark 服务。

### 7.4 平台顺序

1. macOS Apple Silicon：首版正式支持，验证辅助功能投递、全局快捷键、本地模型性能和桌面常驻体验。
2. Windows：核心管线稳定后补齐 UI Automation 和安装升级。
3. Linux：分别评估 X11 与 Wayland，不能预先承诺统一能力。
4. iOS/Android：涉及系统输入法扩展和更严格资源限制，另立规划。

Intel Mac 首版不提供正式安装包，也不承诺性能和兼容性；核心代码保持架构无关，社区可自行构建，后续依据基准决定是否正式支持。

### 7.5 应用更新

- 默认自动检查经过签名的新版本，但不自动下载或安装。
- 发现更新后显示版本、发布日期和发行说明，由用户明确确认下载和安装。
- 未公证的社区构建也允许用户点击后直接下载并更新，不要求跳转浏览器手动替换应用。
- 更新清单和安装包必须使用独立的 Saymore 更新签名；客户端内置公钥，签名无效时拒绝安装。发布私钥不得进入仓库或客户端。
- 社区构建更新后仍可能触发 Gatekeeper，产品必须在确认更新前如实提示；未来加入 Developer ID 签名与公证时复用同一更新协议。
- 更新检查请求不携带稳定设备标识、词典、历史、候选或使用数据，也不能绕过遥测开关承担统计用途。

### 7.6 macOS 发布信任

- Apple Developer Program 账号不是开源 MVP 的开发阻塞项。
- 当前可发布未经过 Apple Notarization 的社区构建；发布页必须明确标记，并提供准确的 Gatekeeper 手动放行说明。
- 当前发布产物仍使用 Saymore 自有更新签名和哈希验证完整性，并允许用户确认后直接更新，但不能称为 Apple 代码签名或公证。
- 取得 Apple Developer Program 账号后，面向普通用户的正式 macOS 安装包必须使用 Developer ID 签名并通过 Apple Notarization。

### 7.7 首发分发渠道

- GitHub Releases 是 MVP 唯一可信发布源；官网只链接到对应的具体 Release。
- 每次发布同时提供 macOS 安装包、应用更新清单、Saymore 更新签名、SHA-256 和发行说明。
- 用户安装包采用 DMG；应用内更新使用更新框架要求的签名归档格式。
- MVP 不提供 PKG，不引入安装脚本，也不要求管理员权限。
- Homebrew Cask 等第三方渠道等安装包格式和更新协议稳定后再支持，不能在首版形成第二套发布真相。

## 8. 体验原则

- **低打扰**：常驻菜单栏，默认显示不含转录文本的状态浮条；用户可关闭并改用提示音与菜单栏图标。
- **入口可选**：登录时启动和 Dock 图标默认开启但可独立关闭，菜单栏作为稳定入口保留。
- **随时可停**：菜单栏可暂停或恢复所有全局录音触发器，不要求退出应用。
- **状态可信**：始终明确当前是录音、识别、精炼还是投递。
- **结果忠实**：系统宁可保留口语，也不能替用户创造内容。
- **内容不丢**：取消以外的失败都应保留已经识别的文字。
- **延迟可感知**：结束口述后明确反馈识别和精炼状态，并尽快一次性投递最终结果。
- **隐私可见**：清楚标明音频和转录文本分别在本地处理还是发送到云端。
- **键盘友好**：开始、结束和取消都可用快捷键完成；内容恢复通过结果浮层和本地历史完成。

## 9. 成功指标

指标先用于工程验收，不作为营销承诺：

- 快捷键到录音状态反馈：p95 小于 100 ms。
- 结束说话到最终文本可投递：本地路径 p95 小于 1.5 s。
- 10 分钟连续使用无丢字、无重复投递、无焦点错投。
- 识别或投递失败时，已有文本恢复率 100%。
- 对中文、中英混合和专业术语测试集持续记录 CER/WER 与热词命中率。
- LLM 精炼测试集持续记录语义保真率、误删率和无依据新增率。
- 断网后本地模式仍能完成“录音到投递”。
- 空闲时 CPU 和内存占用不影响日常前台应用。

## 10. 技术总体方案

### 10.1 架构形态

沿用仓库已有的 Rust 分层，但将 Greeting 示例替换为真实业务边界：

```text
Desktop Shell
  全局快捷键、托盘、可选状态浮条、结果浮层、设置、权限引导
        |
Application
  识别与精炼会话编排、超时、取消、降级与用例
        |
Domain
  识别会话、音频片段、原始转录、精炼文本、投递状态
        |
Infrastructure
  音频、VAD、ASR、LLM、存储、系统凭据、OS 文本投递
```

现有仓库已经使用 Rust + Slint 完成 macOS 辅助功能权限、全局快捷键和文本投递原型。此前的 Tauri/React 界面已在行为一致后删除，项目不维护第二套前端。

### 10.2 建议的 workspace 演进

```text
crates/
  domain/          纯识别领域状态和值对象
  app/             识别用例与端口
  audio/           音频采集、重采样、VAD
  speech/          ASR Provider 与模型生命周期
  text-engine/     安全本地清理、LLM Provider 与精炼策略
  delivery/        跨平台文本投递
  storage/         设置、历史和词典
  cli/             无 UI 的识别诊断和基准工具
  xtask/           模型资源、打包、许可和体积检查
apps/
  desktop/         Slint UI、桌面入口与平台 wiring
```

不建议一开始拆出所有 crate。先保留 `domain/app/infra/cli`，当平台实现或 Provider 出现第二个实现时再提取 `audio/speech/delivery`，避免空抽象。

### 10.3 核心状态机

```text
Idle
 -> Arming
 -> Listening
 -> FinalizingAudio
 -> Transcribing
 -> CleaningTranscript
 -> [Refining]
 -> ReadyToDeliver
 -> Delivering
 -> Completed

任意投递前状态 -> Cancelled（丢弃会话内全部内容）
外部依赖失败 -> RecoverableError -> 已识别文本或本地草稿
```

状态机由 `app` 层拥有。UI 只渲染状态并发送命令，ASR Provider 不得直接操作 UI 或输入框。

### 10.4 关键端口

- `AudioSource`：输出标准化 PCM 帧和音频设备事件。
- `VoiceActivityDetector`：判断语音活动区间，不负责识别内容。
- `SpeechRecognizer`：声明识别能力，接收语言策略和音频流，输出部分与最终转录。
- `TranscriptCleaner`：始终执行不会改变语义的空白、明确 ASR 重复和已确认词典清理。
- `TextRefiner`：在语义保真约束下把最终转录精炼成可投递文本。
- `LlmProvider`：封装一次完整、非流式文本精炼请求、取消和能力信息，不暴露具体供应商或认证方式。
- `TextDeliverer`：验证焦点目标并原子投递文本。
- `TranscriptStore`：只持久化最终输出及其最小元数据，不持久化音频和中间结果。
- `DictionaryStore`：管理已确认词典条目、待确认候选、90 天拒绝冷却和永久屏蔽状态，以及热词筛选所需状态。
- `ModelStore`：管理模型包的下载、校验、导入、删除和本地可用状态。
- `ProviderConfigStore`：在当前用户专属的版本化配置文件中管理 ASR 与自定义 LLM 配置和用户自行提供的密钥；不得记录密钥值。
- `SecretStore`：通过操作系统安全存储管理 Saymore 账户凭据和本地历史数据密钥。

新 trait 要说明职责、流式事件顺序、取消语义和实现方契约。错误类型要区分可重试、可降级、权限缺失和永久失败。

## 11. 技术选型建议

### 11.1 桌面壳

桌面端确定采用 Slint + Rust，不再使用 Tauri、React、TypeScript、Vite、Node.js 或 WebView。设置页、首次引导、历史页、状态浮条和结果浮层共享一套 Slint 组件；业务逻辑、平台集成、异步任务和持久化全部由 Rust 实现。

Slint 的 `.slint` 文件是允许进入仓库的编译期声明式 UI 文件，但不承载业务规则。`domain` 与 `app` 不依赖 Slint；桌面入口只负责把 UI 回调转换成用例命令，并把应用状态映射为 UI view model。

Slint 生成原生桌面程序，但控件是跨平台自绘组件，不等同于 AppKit 或 WinUI 原生控件。macOS 与 Windows 必须分别进行视觉、键盘、屏幕阅读器、缩放和窗口行为验收。

UI 事件循环不得执行音频、ASR、网络、模型加载或数据库阻塞工作。后台任务通过带类型的 Rust 消息把最小状态更新送回 UI 线程。

最大风险是系统集成：macOS Accessibility、Secure Input、焦点切换、沙箱权限和签名发布必须尽早做原型。

打包由 Cargo 与 `xtask` 统一编排，平台工具负责系统要求的签名、公证和安装包创建。macOS 目标产物仍是 `.app` 与 DMG；Windows 安装格式在 Windows 垂直切片中最终确定。完整决策见 `../technology-stack.md` 与 `../adr/0003-use-slint-for-the-desktop-ui.md`。

### 11.2 音频

- 使用 `cpal` 作为跨平台采集入口。
- 内部统一为单声道 16 kHz PCM，避免每个 ASR Provider 自行处理设备格式。
- VAD、重采样、环形缓冲和背压放在独立音频管线中。
- 录音会话必须支持取消，并保证临时音频按策略立即销毁。
- 麦克风热插拔和系统休眠后要能恢复。

### 11.3 ASR

先定义 Provider 契约，再通过基准选择默认实现。候选路径：

- 本地可移植基线：`whisper.cpp` 的 Rust 绑定。
- 中文实时候选：基于 ONNX 的流式模型，例如 Paraformer 或 Zipformer 系列。
- 云端候选：兼容用户自带密钥的流式语音 API。

默认模型不能只按论文准确率决定。必须在目标硬件上比较：

- 中文 CER 和英文 WER。
- 中英切换与专有名词准确率。
- 首字延迟、结束延迟和实时率。
- 内存、CPU、功耗、模型体积和加载时间。
- 流式识别、热词、标点和时间戳能力。
- 模型权重与运行时的开源许可。

MVP 只正式提供一个“均衡”默认模型：

- 最低正式测试设备为 M1、8 GB 内存。
- 必须通过中文、英语、中英混说和语言策略验收。
- 模型下载、内存、CPU 和功耗适合桌面常驻。
- 30 秒以内口述在结束后满足最终转录延迟目标。
- 模型及运行时许可允许开源项目引导下载和使用。
- 默认验证模型的权重许可证必须允许个人与商业本地推理，且不得限制其他合法使用场景；配套运行时代码必须采用 OSI 批准的开源许可证。
- 权重允许再分发不是硬性要求，因为 Saymore 可以引导用户从上游下载；但来源、许可证和再分发条件必须在安装前明确展示。
- 默认模型不能要求注册账号、提交个人信息或额外接受模型专属条款；受门控模型只能作为用户自行配置的扩展，不能成为默认安装路径。
- 多个 Saymore 验证模型档位留到默认模型可靠后再扩展；其他模型先通过自定义 Provider 或手动导入接入。MVP 后的目录、社区和排名规则见 [中文优先 ASR Model Hub](asr-model-hub.md)。

语言能力属于 Provider 契约：

- `supported_languages`：经过该 Provider 验证的语言与地区变体。
- `automatic_language_detection`：是否可以自动识别本次口述语言。
- `intra_utterance_code_switching`：是否支持一句话内切换语言。
- `regional_variants`、`hotwords`、`streaming`、`offline`：其他可选能力。
- 产品提供 `AutoDetect`、`Preferred(language, variant)` 和 `Fixed(language, variant)` 三种策略，并根据能力动态启用。
- 默认选择 `Preferred(系统语言)`；不支持时按 Provider 能力降级。
- LLM 精炼只消费 ASR 判定结果，不修改识别语言声明。

模型与应用分开分发：

- 安装程序不包含模型权重。
- 应用与模型独立版本化；每个 Saymore 版本通过签名的适配清单声明一个推荐默认模型版本和兼容模型范围。
- 适配清单签名只证明 Saymore 对该映射和测试元数据负责，不表示模型由 Saymore 创建或拥有。
- 每个 Saymore 验证模型条目必须展示上游项目、固定版本、原始许可证、下载地址、文件哈希、兼容范围和 Saymore 测试结果。
- 客户端使用内置公钥验证适配清单签名，并校验下载文件的哈希与大小；验证失败时拒绝安装。
- 默认优先从模型作者提供的不可变版本发布地址下载；适配清单可以列出多个镜像，但所有来源必须匹配同一固定哈希。
- Saymore 默认不重新打包或托管第三方模型权重；只有原始许可证明确允许且存在实际可用性需求时，才提供镜像。
- 新安装选择当前应用版本对应的推荐默认模型；这只是默认选择，不把模型版本绑定进应用发布生命周期。
- 应用升级时，当前模型仍兼容则继续使用，并可提示新的推荐模型但不强制切换。
- 当前模型与新应用不兼容时，升级流程必须先准备兼容模型；准备完成前保留旧应用与旧模型可用。
- 首次引导在用户确认后自动下载默认模型，并提前显示名称、大小、许可和预计磁盘占用。
- 下载支持暂停、重试和哈希校验。
- 切换到新模型时默认保留旧模型；新模型至少成功完成一次真实识别后，才提示用户删除旧模型释放空间。
- 新模型加载或识别失败时自动回退旧模型；模型删除始终需要用户确认。
- 用户可删除模型，也可手动导入模型目录用于离线安装和社区模型；未出现在签名适配清单中的模型明确标记为“未由 Saymore 验证”。
- 没有可用模型时允许进入应用设置，但不能开始本地识别。

### 11.4 LLM 文本精炼

- 优先使用 ASR 自带标点、ITN 和热词能力。
- 安全本地清理先处理空白、明确的 ASR 重复和边界可靠的已确认词典替换；不能用正则或词表无条件删除“那个”“然后”“就是”等可能具有语义的词。
- `TextRefiner` 再调用所选 `LlmProvider` 去除口语冗余、处理自我修正、保守润色和整理结构。
- 自定义 LLM 首版支持 OpenAI-compatible Chat Completions 接口，以 `Base URL + API Key + Model` 配置。
- `API Key` 可为空，并支持可选自定义请求头；核心只依赖精炼所需的最小请求、响应、超时、取消和错误契约。
- 本地自定义 LLM 通过同一兼容接口接入，不在核心代码中绑定 Ollama 或 LM Studio 专用 SDK。
- 原生 Anthropic、Gemini 和其他非兼容接口后续各自实现新的 `LlmProvider`。
- 未来的 Saymore 托管 LLM 作为独立 Provider 实现接入。客户端精炼管线不需要改动；登录令牌、额度和服务端数据策略由该 Provider 与后端负责。
- 当前不实现托管 Provider，也不为了它预建计费、账户或同步系统；只确保 `LlmProvider` 不把 API Key 认证写死在应用层。
- 精炼请求只包含安全本地清理后的转录、语言和本次相关的已确认词条，不包含精炼强度、完整个人词典、屏幕、选区、聊天记录或联系人信息。
- ASR 部分结果只保存在当前会话内；`TextRefiner` 只消费最终转录并发起一次非流式请求，不实现增量输入或长文本分块。
- Provider 必须支持取消和失败降级；单次请求超时为 8 秒，当次失败不重试，并返回安全本地清理结果。
- 连续 3 次临时失败触发 5 分钟内存态暂停，成功后清零；认证、模型或配置错误持续旁路，直到配置重新测试成功。
- Provider 原始结果只在当前口述会话内保留，用于降级和诊断；会话结束后不写入历史。
- 使用内置、固定且不可由用户修改的结构化指令、正反示例和输出约束，禁止回答问题、扩写或添加事实。
- 输出检查至少覆盖空结果、异常膨胀、受保护内容变化和非精炼包装内容；检查失败与 Provider 失败使用同一降级路径。

### 11.5 文本投递

macOS 对提供 `AXSelectedTextRange` 的标准可编辑控件优先使用 Accessibility API，并只用写入前后的选区位置验证结果，不读取控件全文或选区原文。光标按写入文本的 UTF-16 长度移动，或控件保持新文本选中，才记为 AX 已验证写入。

AX 明确失败、选区范围始终未变化，或控件不提供可验证范围时，使用临时剪贴板触发 `Command+V`。能观察到光标变化时记为剪贴板已验证；无法读取范围时只记为已尝试，不能冒充成功。若 AX 后的范围发生无法解释的变化，则不再自动粘贴，避免重复输入。无法找到有效输入位置或确认投递失败时，把最终文本交给结果浮层展示和复制。

### 11.6 存储与隐私

- 设置和词典可使用 SQLite；云端 ASR 与自定义 LLM 的用户自备密钥进入当前用户专属的版本化配置文件。
- 本地听写历史默认加密，使用随机数据密钥；数据密钥保存在 macOS Keychain，不能因 Keychain 不可用而静默降级为明文。
- 历史数据密钥丢失或损坏时，停止访问旧历史并明确提示无法解密；用户只能删除旧历史后重新初始化，不提供云端恢复或绕过加密。词典与候选不受影响。
- 个人词典、词汇候选和观察计数不加密，但只保存在本机应用数据目录，不上传。用户主动导出的 JSON 或 CSV 是明文文件。
- ASR 与自定义 LLM 的用户自备 API Key 保存在当前用户专属的版本化配置文件中；macOS 文件权限必须为 `0600`，不得写入日志或听写数据库。Saymore 账户和会话凭据仍保存在 Keychain。
- 本地历史默认开启并保留 7 天；用户可关闭或改为 1 天、30 天或永久保留。
- 缩短保留期限时立即清理已经超期的记录。
- 历史只保存最终输出、时间、投递状态、启用阶段和精炼完成或降级状态，不保存音频、原始转录或 LLM 中间响应。
- 词汇候选与词典条目保存在本地；候选确认前不进入识别或精炼上下文。
- 用于候选发现的原始 ASR 与最终文本差异只在当前会话内计算；会话结束后只保留最小候选映射和计数，不保留完整上下文。
- 候选观察记录按 30 天窗口自动清理；候选进入确认或拒绝状态后删除不再需要的观察计数。
- 原始音频默认不落盘。
- ASR 与 LLM Provider 在设置中分别标识发送的数据、服务地址和数据边界。
- 每个云端 Provider 首次启用前必须显示一次数据发送确认，列出发送的数据、目标服务地址，以及是否包含本次相关的已确认词典条目；用户确认后才能启用。
- 更换 Provider 或 Base URL 后必须重新确认；纯本地 Provider 不需要该确认。未确认词汇候选永远不发送给任何 Provider。
- 匿名使用统计和崩溃报告默认分别关闭，只能由用户主动开启。
- 遥测采用字段白名单；即使用户开启，也禁止上传音频、转录文本、历史、词典、候选、Provider 地址、API Key、请求内容或响应内容。
- Telemetry 默认关闭；崩溃报告发送前可预览。
- 云端 ASR 只发送音频和识别参数；云端 LLM 只发送转录、词典和精炼参数，不发送屏幕或应用内容。

## 12. 首批垂直切片

### Slice 0：可测的识别管线骨架

用文件音频、Fake ASR、Noop TranscriptCleaner、Noop Refiner 和 stdout Deliverer 走通状态机、阶段旁路、取消、超时与恢复。核心逻辑无需桌面 UI 也能确定性测试。

### Slice 1：macOS 任意输入框最小闭环

Slint 已经接入权限状态、右 `Command` 单击切换录音、内存音频采集和单声道 16 kHz signed 16-bit PCM 转换。下一步加入录音浮层与交互声音，再接真实转录结果、Accessibility 最终投递和失败后的结果浮层，优先暴露处理期间焦点变化、目标控件失效和插入失败等平台问题。

### Slice 2：一个可用的本地 ASR

接入一个均衡默认模型，增加签名适配清单、首次下载确认、暂停、重试、哈希校验、手动导入、加载状态、VAD、部分识别和基准命令，并在 M1 8 GB 基线上完成验收。首版只支持这一个 Saymore 验证模型档位。

### Slice 3：用户自定义 LLM

接入 OpenAI-compatible 自定义 Provider，实现单一 LLM 开关、安全本地清理、保守精炼、8 秒超时降级、连续失败暂停、输出检查和语义保真测试集。首版为未来词典接入保留本次相关词条输入，但不等待完整用户记忆功能完成。

### Slice 4：日用可靠性

处理焦点切换、权限撤销、麦克风热插拔、模型损坏、网络超时、剪贴板恢复、崩溃结果恢复和应用升级。

### Slice 5：Provider 可替换

接入第二个本地或云端 ASR，使用同一套 contract tests 验证事件顺序、取消、错误分类和降级行为。

### Slice 6：Windows 闭环

复用识别核心，实现 Windows 音频设备、全局快捷键、UI Automation 投递、安装和升级。

### MVP 后：中文优先 ASR Model Hub

默认本地模型、云端 Provider 边界和日用可靠性均通过验收后，再分阶段增加静态兼容模型目录、公开 benchmark、社区注册表与每日排名快照。Model Hub 不进入上述 MVP 垂直切片，也不能成为本地听写的在线单点故障。完整阶段与安全边界见 [中文优先 ASR Model Hub](asr-model-hub.md)。

## 13. 验证与测试策略

- 领域与 app 层：状态机、取消、8 秒超时、单次调用、降级和幂等投递测试。
- 取消测试：覆盖录音、ASR、安全本地清理和 LLM 精炼阶段，确认不投递、不写历史且清除会话数据。
- Provider contract tests：分别用同一套契约验证 ASR Provider 和 LLM Provider。
- 词汇学习测试：覆盖候选发现、确认、90 天拒绝冷却、永久屏蔽、恢复和删除，并验证未确认候选不能改变识别或精炼结果。
- 词典映射测试：覆盖语言隔离、完整词与短语边界、最长匹配和子字符串误替换。
- 词典导入导出测试：验证 JSON 完整往返、CSV 批量编辑、重复条目合并、冲突阻断，以及候选和拒绝状态不会被导出。
- 加密与凭据测试：验证历史数据库文件不能直接读取用户完整文本、Keychain 密钥缺失时停止访问并允许显式重置而不影响词典，以及 Provider API Key 不进入日志或听写数据库且配置文件权限为 `0600`。
- 词典存储测试：验证词典、候选和观察数据只写入本机应用数据目录，并且不会因同步或遥测上传。
- 遥测测试：验证两个开关默认关闭且相互独立，并用字段白名单阻止用户内容、Provider 配置和凭据进入事件或崩溃报告。
- 更新测试：验证社区构建可以由用户确认后直接更新、无效更新签名被拒绝、私钥不进入产物，以及更新检查请求不包含稳定设备标识或用户数据。
- macOS 发布测试：分别验证未公证社区构建的标识与安装说明，以及未来 Developer ID 构建的代码签名、公证和产物哈希。
- 发布产物测试：验证 GitHub Release 中安装包、更新清单、签名、SHA-256 和发行说明完整且相互对应。
- 安装测试：验证 DMG 拖拽安装、普通用户权限运行，以及应用内更新归档与已安装应用兼容。
- OpenAI-compatible LLM 测试：覆盖 API Key、自定义请求头、无密钥本地服务、非流式响应、8 秒超时、取消、当次不重试和错误映射。
- LLM 连续失败测试：覆盖连续 3 次临时失败后暂停 5 分钟、暂停期间不调用 Provider、到期自动恢复、成功后清零，以及认证、模型或配置错误在重新测试前持续旁路。
- LLM 输出检查测试：覆盖空输出、异常膨胀、网址、邮箱、文件路径、命令、版本号和代码标识符变化，以及模型返回回答或解释时自动降级。
- ASR 能力契约测试：验证声明的语言策略、地区变体、混说、热词、流式和离线行为。
- 语言策略测试：覆盖系统语言偏好、短句歧义、用户切换策略和 Provider 能力降级。
- 音频 fixtures：安静、噪声、停顿、自我修正、中英混说和专业术语。
- 空会话测试：验证无有效语音时不调用下游 Provider、不投递、不写历史、不打开结果浮层，并显示短暂状态。
- 转录 golden tests：完整比较结果，并记录模型、参数和词典版本。
- 精炼 golden tests：覆盖无意义填充词、有语义口语词、口误重复、强调重复、自我修正、语义分段、列表、编号步骤、邮件结构、“换行”与“换一段”的指令和普通语义、中英混说，以及保守换词，检查无依据新增与语义漂移。
- 词典测试：覆盖语言标签、ASR 热词下推、候选筛选、近似匹配和 LLM 请求不包含完整词典。
- 平台集成测试：覆盖原生控件、浏览器、Electron 应用、终端和 IDE。
- 权限引导测试：覆盖麦克风后开始模型下载、下载与后续设置并行、模型校验后才开放外部输入框真实听写、首次投递失败使用结果浮层、跳过后菜单栏继续完成设置、缺失条件只禁用相关能力、辅助功能渐进申请、拒绝后的系统设置入口与重新检测，以及不请求屏幕录制、通讯录或完整磁盘访问。
- Secure 输入测试：覆盖开始录音时拒绝、录音途中切入 Secure 控件后丢弃，以及不写浮层、历史或剪贴板。
- 原子投递测试：验证支持原生撤销的目标应用可用一次 `Cmd+Z` 撤销整段文本。
- 选区投递测试：覆盖替换投递时选区、无选区时插入，以及选区内容不进入任何识别、精炼或遥测数据。
- 结果浮层测试：覆盖复制后关闭、主动关闭、失焦保持和新口述前保存历史。
- 纯文本一致性测试：验证直接投递、结果浮层复制和历史复制得到相同文本，且剪贴板不包含 HTML 或 RTF 表示。
- 仅结果浮层模式测试：覆盖辅助功能未授权、运行中撤销、授权恢复后下一次口述重新直接投递，以及模式内不调用系统投递接口。
- 状态浮条测试：覆盖默认显示、无转录文本、音量反馈、关闭后的提示音与菜单栏反馈。
- 快捷键测试：覆盖默认单一主触发器、直接替换主触发器、“添加另一个”、至少保留一个可用触发器、并发触发器所有权、右 `Command` 捕获验证、键盘冲突、鼠标约束、可录音时消费鼠标事件、不可录音时恢复透传，以及 `Esc` 取消。
- 暂停测试：覆盖空闲暂停、录音中暂停等同取消、全部触发器禁用、鼠标恢复透传、设置仍可访问、跨重启与登录保留，以及菜单栏明确恢复。
- 启动测试：覆盖登录时启动默认开启、用户关闭，以及关闭后手动启动仍能恢复全局触发器。
- 音频设置测试：覆盖自动检测每次跟随系统默认设备、固定设备不可用时不切换、录音中设备断开后保留已有音频且不切换设备、交互声音开关、开始提示音后静音、录音结束立即恢复、处理期间不静音、异常时先恢复再提示，以及录音期间用户主动调节音量时不覆盖新设置。
- 应用行为测试：覆盖 Dock 图标默认显示、关闭后菜单栏入口仍可用、重新开启 Dock 图标、关闭设置或历史窗口不退出，以及菜单栏退出与 `Cmd+Q` 终止进程。
- 活动会话退出测试：覆盖录音中取消后退出、处理中等待或立即丢弃、等待时不直接投递、历史开启时保存后退出、历史关闭时复制或放弃后退出，以及 30 秒超时后继续等待或明确丢弃。
- 本地历史测试：覆盖默认 7 天、关闭历史、期限变更和超期记录立即清理。
- 故障注入：断网、模型卸载、权限撤销、投递时无有效输入位置和进程崩溃。
- 模型切换测试：覆盖新旧模型并存、首次成功后清理提示、加载或识别失败自动回退，以及删除前确认。
- 模型供应链测试：覆盖适配清单签名、固定哈希与大小、无效签名或文件拒绝安装，以及社区模型的未验证标识。
- 模型来源测试：验证多个下载源产物哈希一致，且镜像不可改变清单固定的模型身份与许可证信息。
- 性能基准：首字延迟、结束延迟、CPU、内存、功耗和模型加载时间。
- 长口述测试：覆盖 9 分钟提示、10 分钟自动结束、Provider 更短上限和已有内容完整性。
- 模型管理测试：覆盖确认下载、暂停、重试、校验失败、手动导入和删除。
- 发布门禁继续使用仓库规定的 Cargo fmt/check/nextest/doc/clippy/deny/release/size 流程。

## 14. 主要风险

### 14.1 跨应用投递会直接影响信任

偶发错投、重复粘贴或覆盖剪贴板会让用户立即卸载。平台原型必须早于大规模模型集成。

### 14.2 本地识别与低延迟、低包体存在冲突

高质量本地模型会增加下载、内存和启动时间。需要模型按需下载、硬件分档和云端可选路径。

### 14.3 中英混说与专业词汇准确率不足

通用公开模型未必适合目标用户。必须建立真实测试集，并把个人词典和热词能力作为核心功能，而不是设置页附属项。

### 14.4 最终光标可能不是用户预期的位置

产品明确以实际投递时的光标位置为准，不绑定口述开始位置。用户在处理期间切换窗口或移动光标会改变投递目标；投递时没有可编辑光标则显示结果浮层，避免内容丢失。

### 14.5 模型许可可能限制分发

核心仓库不能分发许可不清晰的权重。模型清单要记录来源、哈希、许可和再分发条件；不允许商业本地推理、限制合法用途或要求门控访问的模型不能成为默认验证模型。

### 14.6 LLM 精炼可能改变用户原意

必须在当前会话内保留原始转录用于降级、限制输入上下文、使用固定保守提示词并检查输出，针对事实新增、异常膨胀、数字变化、否定词丢失、受保护内容变化和专有名词误改建立回归测试。LLM 失败或结果被拒绝时自动回退到安全本地清理结果，不能成为输入链路的单点故障。

### 14.7 Model Hub 可能稀释核心产品

多模型适配、社区审核、排名、遥测和镜像会把桌面工具扩展为持续运营的平台。MVP 不建设这些能力；只有默认本地模型、核心听写和失败恢复达到日用可靠性后，才按独立路线推进。模型数量、下载量和社区活跃不能替代听写成功率、延迟与用户留存。

## 16. 推荐的下一步

桌面技术栈已经确定为 Slint + Rust。下一步按以下顺序推进：

1. **macOS 投递矩阵**：验证 10 个目标应用中的焦点识别、原子文本写入、原生撤销和失败后的结果浮层。
2. **ASR 基准**：使用同一批中文、中英混说和术语音频，对候选模型或云端 Provider 测准确率、延迟、资源占用、体积、价格和许可。
3. **LLM 精炼保真实验**：用同一批真实转录比较多个自定义 LLM，测语义保真、误删、无依据新增、延迟和失败降级。

后三项结果决定首版 Provider 和产品承诺，不再改变已经确定的桌面 UI 技术栈。之后再把后续垂直切片转成可独立实现和验收的开发任务。
