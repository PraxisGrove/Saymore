# 闪电说与 Typeless：桌面端 Onboarding 调研

调研日期：2026-07-16。优先采用官网、官方帮助中心和官方产品页。下文将“官方明确说明”与“根据官方截图/页面结构推断”分开，避免把推断当成产品事实。

## 结论先行

两家的共同点不是页面长得像，而是都把 onboarding 当成一条“完成首次成功输入”的任务链：解释价值只是开场，权限、输入设备和快捷键必须实际跑通，最后让用户亲眼看到语音变成整理后的文字。

Saymore 首版适合借鉴 Typeless 的短闭环：

1. 欢迎与一句话价值说明。
2. 麦克风权限，并显示实时音量反馈。
3. 辅助功能权限，解释其用于全局快捷键和将文字写入当前输入框。
4. 确认或设置快捷键。
5. 在内置练习框完成一次真实语音输入。
6. 完成页说明 Saymore 会在后台运行，以及去哪里打开、暂停和退出。

不要把账号、付费、词典、模型高级配置和全部设置塞进首次引导。Saymore 当前没有账号系统，闪电说和 Typeless 的登录步骤没有照搬价值。

## 闪电说

### 官方明确说明的流程

[官方入门指南](https://shandianshuo.cn/docs)明确将首次上手压缩为三件事：“模型跑通、短按能出字、长按能帮你说”。完整顺序是：

1. 先让模型可用：登录领取试用、购买会员，或自己配置 ASR/LLM 提供商。
2. 在微信、飞书、备忘录、Word 或网页输入框中，用短按快捷键完成一次语音输入。
3. 用长按快捷键体验结合屏幕上下文生成回复。
4. 后续才引导技能、词典和长期表达偏好。
5. 遇到问题时检查麦克风、辅助功能和屏幕录制权限；官方同时说明 app 内 onboarding 会逐步引导这些权限。

权限用途也写得很具体：麦克风用于录音和识别；辅助功能用于监听快捷键并把文字写进当前输入框；屏幕录制用于高级的“帮我说”读取屏幕上下文。

### 视觉与交互模式

- **官方明确说明**：首页顶部持续展示当前快捷键，并提供“查看演示”，用来对比短按与长按两种动作。
- **官方明确说明**：模型状态以可用状态呈现，第三方服务商可用时显示绿色圆点。
- **从官方文档截图及其说明推断**：界面倾向将当前关键动作放在首页上方，并用状态标识告诉用户“这一环是否已跑通”，而不是只给一页文字教程。

官方公开资料没有完整逐屏展示 app 内 onboarding，因此无法确认欢迎页数量、进度条样式、每个权限是否独占一页等细节。

### Saymore 可借鉴什么

- 把“用户完成一次真实输入”定义为 onboarding 的成功，而不是点击完所有“下一步”。
- 在引导与设置中始终显示用户当前实际快捷键，不把固定快捷键写死在插画里。
- 权限解释采用“为什么需要 + 开启后能做什么”的具体语言。
- 高级能力后置，先跑通最基础的语音输入闭环。

### 不该照搬什么

- 不照搬“先登录/购买/配置三类模型”。这是闪电说当前商业和技术架构产生的步骤，Saymore 首次使用不应暴露供应商复杂度。
- 不照搬短按/长按双模式，除非 Saymore 已经明确拥有两个稳定且不同的核心动作。
- 不要在首次引导中加入技能、词典和个性化完整教学；可在首次成功后用轻量提示渐进发现。

## Typeless

### 官方明确说明的流程

[官方安装与设置指南](https://www.typeless.com/help/installation-and-setup)给出了较完整的桌面端顺序：

1. 下载、安装并启动。
2. 使用 Google 或邮箱登录。旧版私测流程还包含邀请码激活；[官方版本说明](https://www.typeless.com/help/release-notes/macos)显示候补名单和邀请码后来已经取消。
3. 开启辅助功能权限和麦克风权限。
4. 选择麦克风并说话；看到蓝色音量条移动后确认继续。
5. 测试语音快捷键。macOS 默认 `Fn`，Windows 默认 `Right Alt`，且允许自定义。
6. 在引导中朗读示例，依次体验列表自动排版、邮件格式化、口误自动修正和低声识别。
7. 完成后进入任意应用输入框使用。

[官方首次听写指南](https://www.typeless.com/help/quickstart/first-dictation)把核心动作压缩为四步：点入任意文本框、按一次快捷键、自然说话、再按一次快捷键完成并插入整理后的文字。

### 权限与快捷键教学

- 辅助功能权限先于麦克风权限；官方解释它用于将文字插入当前输入框，并支持全局快捷键。
- 权限申请前先显示 Typeless 自己的解释页，再触发系统授权弹窗。
- 麦克风授权后不是直接结束，而是用实时音量条验证设备确实工作；多设备时允许选择输入源。
- 快捷键不是静态介绍，而是让用户当场按键并完成练习。

### 视觉与交互模式

- **从官方帮助中心所附产品截图推断**：每一步聚焦一个任务，中央是大标题、简短解释和单一主操作，减少同时出现的选择。
- **官方明确说明**：麦克风测试使用蓝色动态音量条作为即时反馈。
- **官方明确说明**：效果体验使用可朗读的具体样例，并直接呈现整理后的结果，例如口述购物清单变成项目符号列表。
- **从官方截图序列推断**：整体是线性、可验证的 setup，而不是在首页叠加多个教学浮层。

### Saymore 可借鉴什么

- 一项系统权限一页，并在触发系统弹窗之前解释用途。
- 权限状态要能被检测：授权成功自动进入下一步；拒绝时保留“打开系统设置”和重试入口。
- 麦克风页提供实时电平，不只显示“已授权”。授权成功和硬件真正可用是两件事。
- 让用户亲手按快捷键，并在内置输入框完成一次真实转写；这同时验证麦克风、快捷键、ASR 和文字输出链路。
- 用一个能显示 AI 整理价值的短示例，而不是长篇功能轮播。

### 不该照搬什么

- 不照搬登录和激活步骤；Saymore 没有账号系统时，这只会制造无意义阻力。
- 不需要一次做四个能力演示。首版只选最能体现 Saymore 差异的一个示例。
- 不应直接采用 `Fn` 作为默认键而不验证 macOS 系统、键盘型号及现有快捷键冲突。
- 不要让用户在首次引导里配置大量偏好；先成功，再个性化。

## 给 Saymore 的首版页面建议

| 页面 | 目的 | 主要操作 | 成功条件 |
| --- | --- | --- | --- |
| 1. 欢迎 | 说明“在任何输入框中说话即可输入” | 开始设置 | 用户明确进入设置流程 |
| 2. 麦克风 | 解释录音用途并验证设备 | 允许麦克风；选择设备（必要时） | 已授权且音量条有输入 |
| 3. 辅助功能 | 解释全局快捷键和文字插入 | 打开系统设置/允许 | 系统检测为已授权 |
| 4. 快捷键 | 教会启动与停止录音 | 按下当前快捷键；可更改 | 捕获到一次合法快捷键动作 |
| 5. 第一次输入 | 让用户真实感受结果 | 在练习框中说一句短文本 | 转写成功并显示结果 |
| 6. 完成 | 解释后台常驻入口 | 完成 | 记住 onboarding 已完成 |

设计上建议保持一个固定尺寸窗口、清晰的步骤进度、每页一个主按钮。权限页应始终显示三态：未开启、已开启、需要修复。用户拒绝授权时不要陷入死路，也不要假装已经成功。

完成页需要直接说明平台差异：macOS 从屏幕顶部状态栏图标打开、暂停或退出 Saymore；Windows 从右下角系统托盘图标操作。它不是 macOS Dock 右键菜单，也不是 Windows 任务栏窗口按钮。

## 仍需产品确认的问题

1. Saymore 首版是否只要求麦克风和辅助功能；若没有读取屏幕的功能，不应申请屏幕录制权限。
2. 用户能否跳过辅助功能权限并只在主窗口内录音；若不能，必须明确说明此权限是核心前提。
3. 首次练习是否消耗云端额度；如果会，应在发起请求前清楚说明。
4. onboarding 完成后是否默认启用“登录时启动”。建议明确询问或在完成页提供复选项，不要静默开启。
5. 重新打开 onboarding 的入口放在“帮助”还是“设置”。建议设置中提供“重新运行新手引导”。
