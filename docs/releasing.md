# Releasing

Saymore releases are built by `.github/workflows/release.yaml`. The workflow is
started manually from the GitHub Actions page and proceeds only when `main` has
commits newer than the latest GitHub Release tag. This keeps ordinary pushes and
small changes covered by CI without automatically publishing a new stable
version.

This document currently covers direct distribution through GitHub Releases. The
planned Mac App Store submission and release workflow will be documented
separately when its packaging and review requirements are implemented.

GitHub supplies the latest release tag and commit comparison; the tested
`cargo run -p xtask -- release-plan` command owns the deterministic eligibility
and version decision.

## Versioning

The workspace version in the root `Cargo.toml` is the only application version
source. The first automated release uses that version. Later releases increment
the patch component of the latest `vMAJOR.MINOR.PATCH` GitHub Release tag.

Each packaging job applies the planned version before compiling. After every
package succeeds, the publish job commits the version change when needed,
creates an annotated tag, pushes both atomically, and creates the GitHub
Release. The macOS `CFBundleVersion` is a monotonically increasing UTC build
timestamp generated by `cargo-packager`.

The release plan requires a successful `CI` push run for the exact source commit
before packaging starts. CI checks, tests, and lints all workspace targets on
both macOS and Windows, then builds the production desktop binary in release
mode. Dependency policy runs separately on Ubuntu.

## Artifacts

- macOS builds both Apple Silicon and Intel targets, combines them into one
  universal binary, and publishes a branded drag-to-Applications DMG plus
  SHA-256 checksums to GitHub Releases. The repository-owned DMG background,
  window size, and icon positions are declared in the desktop package metadata.
  The DMG opens directly to its Finder window without an install-time license
  agreement. With complete Apple credentials, packaging signs and notarizes the
  application, signs the final DMG, submits that DMG to Apple, staples the
  accepted ticket, and requires Gatekeeper to accept it before upload. The
  workflow then mounts the generated image without accepting an EULA and
  verifies its background, Finder layout metadata, application bundle, and
  Applications link. Regenerate the committed background after design changes
  with `swift apps/desktop/packaging/macos/generate-dmg-background.swift`.
  Without complete Apple credentials, the DMG filename ends in `-unsigned.dmg`.
- Windows builds an NSIS installer named `Saymore-Setup.exe` and publishes it,
  together with its SHA-256 checksum, to the same GitHub Release.

Linux is not built or published until its platform adapters are implemented and
the Linux target is explicitly restored to CI and the release workflow.

## Required Repository Configuration

Set the workflow permission to allow GitHub Actions to write repository
contents. The release commit also requires `main` branch rules to permit the
GitHub Actions bot to push.

Apple credentials are optional for running the release workflow. Without them,
the workflow publishes an unsigned and unnotarized DMG; macOS users must use the
Finder context menu to open it and accept the security warning. For normal
click-to-open distribution, join the Apple Developer Program and add these
GitHub Actions secrets:

- `APPLE_CERTIFICATE`: base64-encoded Developer ID Application `.p12` file.
- `APPLE_CERTIFICATE_PASSWORD`: password used when exporting the `.p12` file.
- `APPLE_ID`: Apple account used for notarization.
- `APPLE_PASSWORD`: app-specific password for that account.
- `APPLE_TEAM_ID`: Apple Developer team identifier.

All five secrets must be present to enable signing. A partial configuration
falls back to the explicitly labelled unsigned DMG.

## Recovery

The publish step is idempotent. If the version tag was pushed but creating or
uploading the GitHub Release failed, rerun the same workflow. It rebuilds from
the tagged commit, then creates or repairs the Release assets.
